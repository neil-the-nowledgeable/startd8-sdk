The Problem

  • Documentation describes intent, code reflects what was built — they drift apart
  • Sales promises capabilities that are still draft; users expect them to work
  • Engineers re-discover constraints that were already known but never recorded
  • AI agents can't discover capabilities without loading entire schemas (600+ tokens)
  • Maturity is guessed, not measured — "beta" and "stable" mean different things to everyone

The Capability Index

  • Multi-audience descriptions — each stakeholder gets their own truth
  • Maturity-gated messaging prevents vaporware; draft capabilities can't claim stability
  • Intent traceability captures why things were built, even when they pivot
  • Token-efficient progressive disclosure — 95% reduction in discovery overhead
  • Evidence-grounded claims — every capability links to code, tests, or docs

Core Principles

1 Perspectival Truth

Different audiences have valid different views of the same capability. A user sees "I can secure my account." An engineer sees "RFC 6238 TOTP in src/auth/mfa/." Sales sees "enterprise compliance checkbox." None is wrong. Each is incomplete.

2 User Benefit is the Root

Every capability traces to user value — directly or through dependencies. Infrastructure capabilities must justify themselves via what they enable. Orphan capabilities are suspect.

3 Humans Author, Machines Propagate

The index is written by humans with judgment. Machines generate derived views (MCP tools, API docs, agent cards) from the canonical YAML. Humans stay in control of meaning.

4 Evidence Over Assertion

Claims must have verifiable references. Higher confidence requires stronger evidence. The evidence field connects assertions to artifacts — code, tests, ADRs, traces.

5 Intent Preserved

When plans change, the index captures why. Abandoned intent is still valuable knowledge. This prevents re-learning lessons, mis-promising pivoted features, and silent documentation lies.

6 Gaps Made Visible

Different perspectives will diverge. The index makes divergence explicit rather than hidden. Benefits with empty delivered_by fields are gaps — they become roadmap items automatically.

Tooling: Validate, Query, Generate

validate.py

Schema validation with evidence checking. Catches broken references before they reach users.

python3 tools/validate.py manifests/otel/otel.framework.yaml

query.py

Filter by ID, category, maturity, audience, trigger, or confidence. Prefix and substring matching.

python3 tools/query.py manifests/ --maturity stable --audience agent

generate_mcp.py

Generate MCP tool definitions from capabilities. One command, and agents can discover your system.

python3 tools/generate_mcp.py manifests/startd8.yaml --output tools.json

generate_a2a.py

Generate A2A Agent Cards for cross-agent discovery. Follows the .well-known convention.

python3 tools/generate_a2a.py manifests/startd8.yaml --well-known

extract_capabilities.py

Auto-extract capabilities from any codebase. Phase 1 of the 3-phase user-facing workflow.

python3 tools/extract_capabilities.py /path/to/project --project-name "MyApp"

aggregate.py

Pull manifests from multiple repos for a unified cross-service view. Hybrid deployment model.

python3 tools/aggregate.py --config sources.yaml --output full-index.yaml

Get Started

# Validate an existing manifest
python3 tools/validate.py manifests/startd8.workflow.capabilities.yaml

# Query stable capabilities for agents
python3 tools/query.py manifests/ --maturity stable --audience agent --verbose

# Extract capabilities from your own project
python3 tools/extract_capabilities.py /path/to/project --project-name "MyProject"

# Generate MCP tools for agent discovery
python3 tools/generate_mcp.py manifests/startd8.yaml --output startd8-tools.json

# Or use the StartD8 CLI directly
startd8 workflow list
startd8 workflow describe pipeline
startd8 workflow run pipeline --dry-run
