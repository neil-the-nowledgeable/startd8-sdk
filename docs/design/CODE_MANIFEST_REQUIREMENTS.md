## 1. Vision

Leverage Python's runtime execution model to generate a **digital manifest** of source code — a canonical, deterministic, programmatically-readable map of every structural element in a Python file. This manifest enables surgical code modification, blast-radius estimation, and ground-truth validation across the startd8 pipeline (capability index, plan ingestion, artisan workflow).

### 1.1 Problem Statement

Today the artisan pipeline treats Python files as opaque text blobs:

- **IMPLEMENT** generates entire files, destroying existing production code (Lesson Leg 13 #29)
- **Plan ingestion** extracts API signatures via regex heuristics, missing structure
- **Preflight validators** re-parse AST per check with no shared structural index
- **Code review** cannot reason about cross-file relationships or change blast radius
- **Capability index** catalogs capabilities declaratively with no source-derived ground truth

### 1.2 Goal

Produce a structured manifest from Python source that:

1. Canonically maps every addressable element (class, function, variable, import) with human-readable names and machine-resolvable coordinates
2. Enables deterministic identification and surgical updating of file portions
3. Informs downstream pipeline stages (plan ingestion, artisan workflow, code review) about existing code structure
4. Can be generated by the capability index build command and consumed across the pipeline

## 2. Python Execution Model Layers

Python exposes its execution model at five granularities, each yielding different manifest data. These layers are ordered from static to dynamic — earlier layers require no code execution, later layers require import or runtime tracing.

### 2.1 Layer 1: AST (Abstract Syntax Tree) — Static Structure

**Source:** `ast.parse()` from the `ast` standard library module

**What it provides:**
- Complete syntax tree of a Python file without executing it
- Every `ClassDef`, `FunctionDef`, `AsyncFunctionDef`, `Assign`, `AnnAssign`, `Import`, `ImportFrom`, decorator, and control flow node
- Exact source positions: `lineno`, `col_offset`, `end_lineno`, `end_col_offset` on every node
- Deterministic — same source always produces the same tree
- Round-trippable via `ast.unparse()` (Python 3.9+)

**Manifest contribution:**
- Element kind, name, fully-qualified name (FQN), source span
- Function/method signatures (parameter names, annotations, defaults)
- Class hierarchy (base classes, decorators)
- Import graph (what modules/names are imported)
- Constant definitions and type annotations
- Docstrings (first `Expr(Constant(str))` in body)

**Limitations — Comment and Formatting Preservation:**

`ast.parse()` discards comments, blank lines, and original formatting. `ast.unparse()` produces syntactically valid but reformatted code. This means:

- The manifest accurately captures structural elements and spans, but the AST alone cannot round-trip comments or formatting.
- **Surgical edits must use span-based text splicing**, not AST unparse. The manifest provides `[start_line, start_col, end_line, end_col]` spans that identify the exact character range of each element in the original source. Edit operations replace the text within a span, preserving all surrounding content including comments and formatting.
- For cases requiring AST-level manipulation (e.g., signature modification), implementations should consider CST (Concrete Syntax Tree) libraries such as `libcst` that preserve comments, whitespace, and formatting. The manifest's span data remains the source of truth for element boundaries regardless of the edit tooling used.

**Limitations — Dynamically Generated Code:**

Static AST analysis cannot detect or represent code generated at runtime through patterns such as `exec()`, `eval()`, `type()` used as a class factory, `collections.namedtuple()`, `dataclasses.make_dataclass()`, or methods injected by metaclasses. These patterns produce no AST nodes at the point of definition. They are **explicitly out of scope for static mode**. If a variable is assigned the result of such a call (e.g., `Point = namedtuple('Point', ['x', 'y'])`), the assignment itself appears in the manifest as a `variable` element, but the generated class's methods and attributes do not. Runtime introspection (Layer 3, P2) may partially recover this information for modules that can be safely imported.

**Priority:** P0 — this is the foundation layer. All other layers augment it.

### 2.2 Layer 2: Symbol Table — Scope and Binding

**Source:** `symtable.symtable()` from the `symtable` standard library module

**What it provides:**
- Which names are local, global, nonlocal, imported, or free (closure) variables
- Scope nesting (module → class → function → nested function)
- Whether a name is referenced, assigned, or both
- Whether a symbol is a parameter

**Manifest contribution:**
- Variable scope classification (local / global / nonlocal / free / imported)
- Read vs. write analysis per scope
- Closure variable identification
- Unused name detection

**Priority:** P1 — augments AST with binding semantics not visible from tree structure alone.

### 2.3 Layer 3: Runtime Introspection — Live Object Metadata

**Source:** `inspect` module (`getmembers`, `signature`, `getsource`, `getmro`)

**What it provides:**
- Resolved type annotations (including forward references)
- Default values evaluated at import time
- Method Resolution Order (MRO) for classes
- Actual callable signatures (accounting for `*args`, `**kwargs`, decorators)
- Module-level `__all__`, `__version__`, and other dunder attributes

**Manifest contribution:**
- Resolved signatures (when AST annotations are string forward-refs)
- Class MRO chain
- Public API surface (from `__all__`)
- Runtime-computed defaults

**Constraints:** Requires importing the module, which may trigger side effects. Must be opt-in.

**Priority:** P2 — valuable for API surface mapping but requires controlled execution.

### 2.4 Layer 4: Bytecode — Instruction Stream

**Source:** `dis` module (`dis.dis()`, `dis.Bytecode()`, `dis.get_instructions()`)

**What it provides:**
- The actual CPython bytecode instructions: `LOAD_FAST`, `CALL_FUNCTION`, `STORE_NAME`, etc.
- Call graph edges that static analysis misses (dynamic dispatch, `getattr`, computed attribute access)
- Data flow between variables within a function

**Manifest contribution:**
- Intra-function call graph (what functions/methods a function calls)
- Attribute access patterns (`self.x`, `obj.method()`)
- Dynamic name resolution detection (flags elements that resist static analysis)

**Constraints:** CPython-specific. Bytecode format changes between Python versions.

**Priority:** P3 — primarily useful for call graph and data flow; defer unless needed.

### 2.5 Layer 5: Execution Trace — Dynamic Path Coverage

**Source:** `sys.settrace()`, `sys.setprofile()` (same mechanism as `coverage.py`)

**What it provides:**
- Which code paths actually execute for a given input
- Function call/return events with arguments
- Line-level execution counts

**Manifest contribution:**
- Dead code identification (elements never reached in test suite)
- Hot path analysis
- Runtime type inference (observed argument types)

**Constraints:** Requires executing the code with representative inputs. Non-deterministic.

**Priority:** P4 — future enhancement for test-coverage-informed manifests.

## 3. Manifest Schema

### 3.1 File-Level Manifest

Each Python file produces one manifest document containing:

| Field | Type | Description |
|-------|------|-------------|
| `schema_version` | `string` | Manifest schema version (semver, e.g., `"1.0.0"`). See Section 3.5 for versioning contract. |
| `file` | `string` | Relative path from project root |
| `module` | `string` | Python module path (dot-separated) |
| `digest` | `string` | Content hash (`sha256:...`) for staleness detection |
| `python_version` | `string` | Minimum Python version required (from AST features used) |
| `elements` | `list[Element]` | Ordered list of top-level and nested structural elements |
| `imports` | `list[Import]` | All import statements |
| `dependencies` | `Dependencies` | Classified dependency summary |
| `errors` | `list[ParseError]` | Parsing errors encountered during manifest generation (empty list if none). See Section 5.7. |

The `ParseError` schema is:

| Field | Type | Description |
|-------|------|-------------|
| `kind` | `enum` | `syntax_error`, `encoding_error`, `io_error`, `partial_parse` |
| `message` | `string` | Human-readable error description |
| `line` | `int?` | Line number where the error occurred, if available |
| `col` | `int?` | Column offset where the error occurred, if available |

### 3.2 Element Schema

Every addressable code element shares a common base:

| Field | Type | Description |
|-------|------|-------------|
| `kind` | `enum` | `class`, `function`, `async_function`, `method`, `async_method`, `property`, `constant`, `variable`, `type_alias` |
| `name` | `string` | Human-readable name (e.g., `run_phase`) |
| `fqn` | `string` | Fully-qualified name (e.g., `startd8.contractors.artisan_contractor.ArtisanContractorWorkflow.run_phase`) |
| `span` | `Span` | Source location: `{start_line, start_col, end_line, end_col}` |
| `docstring` | `string?` | First string literal in body, if present |
| `decorators` | `list[string]` | Decorator expressions as full source text, preserving arguments and attribute access. For example: `"staticmethod"`, `"app.route('/users', methods=['POST'])"`, `"dataclass(frozen=True)"`. The leading `@` is omitted. Each entry is the exact source text of the decorator expression. |
| `children` | `list[Element]` | Nested elements (methods in class, nested functions) |
| `scope_guard` | `string?` | Conditional context in which this element is defined, if any. `null` (or absent) when the element is defined unconditionally at its scope level. Values include `"__main__"`, `"TYPE_CHECKING"`, `"try"`, `"except"`, or a descriptive string of the condition (truncated to 80 characters). See Section 4.1.1 for the full rules governing this field. |

**Note on `kind` and type aliases:** The `type_alias` kind is used for Python 3.12+ `type` statements (e.g., `type Vector = list[float]`). For pre-3.12 type alias patterns using `TypeAlias` annotation (e.g., `Vector: TypeAlias = list[float]`), the element kind is `variable` with the `type_annotation` field set to `"TypeAlias"`. This distinction reflects the difference in the AST: `type` statements produce a dedicated AST node, while annotated assignments are syntactically assignments. The `is_type_alias` field in Section 3.2.3 is removed to avoid ambiguity; the kind enum is the sole discriminator.

#### 3.2.1 Callable Elements (function, method, async variants)

Additional fields for callables:

| Field | Type | Description |
|-------|------|-------------|
| `signature` | `Signature` | Parameters and return type |
| `signature.params` | `list[Param]` | `{name, annotation?, default?, kind}` where kind is `positional`, `keyword`, `var_positional`, `var_keyword` |
| `signature.return_annotation` | `string?` | Return type annotation |
| `is_static` | `bool` | Has `@staticmethod` decorator |
| `is_classmethod` | `bool` | Has `@classmethod` decorator |
| `is_abstract` | `bool` | Has `@abstractmethod` decorator |
| `visibility` | `enum` | `public`, `protected` (single `_`), `private` (double `__`) |
| `overload_index` | `int?` | Zero-based index among `@typing.overload` variants for this name. `null` (or absent) for non-overloaded callables and for the implementation definition of an overloaded name. See the `@typing.overload` handling rules below. |

**Handling `@typing.overload` Definitions:**

Functions or methods decorated with `@typing.overload` represent type-level signature variants, not runtime implementations. The manifest represents these as follows:

- Each `@overload`-decorated definition is included as a separate element with an `overload_index` field (`int`, zero-based) indicating its position among overload variants.
- The implementation definition (the non-`@overload` version with the same name) has `overload_index: null` (or the field is absent).
- All overloads and the implementation share the same `name` and base `fqn`. Overloaded variants are disambiguated by appending `@overload[N]` to the FQN, where `N` is the zero-based index. For example:
  - `module.MyClass.process@overload[0]` — first overload
  - `module.MyClass.process@overload[1]` — second overload
  - `module.MyClass.process` — implementation
- The `decorators` list for each overload includes `"typing.overload"` (or `"overload"` depending on how it was imported).

**Handling `@property` Getter/Setter/Deleter Triads:**

A `@property` and its associated `@name.setter` and `@name.deleter` definitions share the same name within a class. The manifest represents these as follows:

- The `@property` getter is represented as kind `property` with its own span and signature.
- The `@name.setter` and `@name.deleter` are represented as separate elements with kind `method`, appearing as children of the class.
- Each is disambiguated by appending an accessor suffix to the FQN:
  - `module.MyClass.value@getter` — the `@property` definition
  - `module.MyClass.value@setter` — the `@value.setter` definition
  - `module.MyClass.value@deleter` — the `@value.deleter` definition
- The `decorators` list accurately reflects the actual decorator (`"property"`, `"value.setter"`, `"value.deleter"`).

#### 3.2.2 Class Elements

Additional fields for classes:

| Field | Type | Description |
|-------|------|-------------|
| `bases` | `list[string]` | Base class names/expressions |
| `metaclass` | `string?` | Metaclass if specified |
| `tags` | `list[string]` | Extensible list of framework and pattern tags. Auto-populated by heuristic detection; values include `"pydantic_model"` (inherits from `BaseModel` or has `model_config`), `"dataclass"` (has `@dataclass` or `@dataclasses.dataclass` decorator), `"attrs"`, `"django_model"`, `"protocol"`, `"abstract"`, etc. New tags can be added without schema changes. |
| `class_variables` | `list[Element]` | Class-level assignments |

The `tags` field replaces per-framework boolean flags (which would require schema changes for every new framework), providing an open vocabulary instead. Detection heuristics for tag assignment:

- `"pydantic_model"`: class inherits from `BaseModel`, or has a `model_config` class variable, or uses `@validator`/`@field_validator` decorators.
- `"dataclass"`: class has `@dataclass` or `@dataclasses.dataclass` decorator.
- `"attrs"`: class has `@attr.s`, `@attrs`, or `@define` decorator.
- `"protocol"`: class inherits from `typing.Protocol`.
- `"abstract"`: class inherits from `ABC` or has any `@abstractmethod` methods.

Additional tags may be defined by convention without requiring schema version changes.

#### 3.2.3 Assignment Elements (constant, variable)

Additional fields for assignments:

| Field | Type | Description |
|-------|------|-------------|
| `type_annotation` | `string?` | Type annotation if present |
| `value_repr` | `string?` | Short string representation of assigned value. See representation rules below. |

**`value_repr` Representation and Truncation Rules:**

To ensure deterministic output, `value_repr` follows these rules:

1. **Simple literals** (int, float, bool, None, short strings): represented as their source text. Example: `42`, `True`, `"hello"`, `None`.
2. **String literals**: represented in full if ≤ 80 characters; otherwise truncated to 77 characters followed by `...`. Example: `"This is a very long string that exceeds the limit..."`.
3. **Collection literals** (list, dict, set, tuple): represented in full if the source text is ≤ 120 characters; otherwise truncated to 117 characters followed by `...`. Example: `[1, 2, 3]`, `{"key": "value", ...}`.
4. **Function calls and complex expressions**: represented as the source text of the expression, truncated at 120 characters with `...` if exceeded. Example: `SomeClass(param1="value", ...)`.
5. **Multi-line values**: collapsed to a single line with internal whitespace normalized to single spaces before applying truncation rules.
6. **Maximum absolute length**: 120 characters including the truncation marker. No `value_repr` shall exceed this length.
7. **Null case**: if the value cannot be meaningfully represented (e.g., deeply nested expression), `value_repr` is `null`.

### 3.3 Import Schema

| Field | Type | Description |
|-------|------|-------------|
| `kind` | `enum` | `import`, `from` |
| `module` | `string` | Resolved absolute module path (dot-separated). For relative imports (e.g., `from ..utils import bar`), this field contains the resolved absolute path (e.g., `"startd8.utils"`), computed using the file's position within the project and its package hierarchy. If the relative import cannot be resolved (e.g., the project root is unknown or the relative path escapes the package), the raw relative syntax is stored (e.g., `"..utils"`) and a warning is emitted in the file-level `errors` field. |
| `names` | `list[string]` | Imported names (for `from` imports) |
| `alias` | `string?` | Alias if `as` used |
| `span` | `Span` | Source location |
| `is_relative` | `bool` | Whether the original import used relative syntax (e.g., `from . import foo`, `from ..utils import bar`). This field preserves provenance information that is lost when the `module` field is resolved to an absolute path. Always `false` for absolute imports. |
| `is_conditional` | `bool` | Inside `if TYPE_CHECKING:` or `try/except ImportError` |
| `is_reexport` | `bool` | Whether this import constitutes a re-export of the name. An import is classified as a re-export if any of the following heuristics match: (a) **`__init__.py`-specific**: it appears in an `__init__.py` file and imports from a submodule of the same package, or (b) **Universal (all files)**: the imported name appears in the module's `__all__` list — this heuristic applies to any Python file, not just `__init__.py`, or (c) **`__init__.py`-specific**: the import is a `from .submodule import Name` pattern in an `__init__.py` without aliasing or further use beyond exposure. Heuristics (a) and (c) are specific to `__init__.py` files, while heuristic (b) applies universally to any file that defines `__all__`. This field enables accurate public API surface mapping for packages. |

### 3.4 Dependency Summary

| Field | Type | Description |
|-------|------|-------------|
| `internal` | `list[string]` | Intra-project module dependencies |
| `external` | `list[string]` | Third-party package dependencies |
| `stdlib` | `list[string]` | Standard library dependencies |
| `conditional` | `list[string]` | Optional/guarded dependencies |

**Dependency Classification Algorithm:**

Dependencies are classified using the following rules, applied in order:

1. **Standard library**: A module is classified as `stdlib` if it appears in the known stdlib module list for the target Python version. The stdlib list is derived from `sys.stdlib_module_names` (Python 3.10+) or a vendored snapshot for earlier versions. This includes top-level package names only (e.g., `os`, `json`, `collections`).

2. **Internal (intra-project)**: A module is classified as `internal` if:
   - It is a relative import (starts with `.`), or
   - Its top-level package name matches the project's root package (determined from the project root path and `src/` layout convention), or
   - It resolves to a file within the project root directory tree.

3. **External (third-party)**: A module is classified as `external` if it is neither stdlib nor internal. The top-level package name (the first component of the dotted path) is recorded, not the full module path. For example, `from pydantic.fields import Field` records `pydantic` as the external dependency.

4. **Conditional**: Any dependency imported inside an `if TYPE_CHECKING:` guard, a `try/except ImportError` block, or a function body is additionally tagged as `conditional`. A conditional dependency also appears in one of the above three categories based on its module classification.

Note: Classification accuracy depends on knowledge of the project root and the Python version's stdlib. When these cannot be determined, the generator should emit a warning in the `errors` field and default ambiguous modules to `external`.

### 3.5 Schema Versioning

The manifest schema follows semantic versioning (`MAJOR.MINOR.PATCH`):

- **MAJOR** version increments indicate breaking changes: fields removed, field types changed, or field semantics redefined. Consumers must be updated to handle a new major version.
- **MINOR** version increments indicate additive changes: new optional fields, new enum values in `kind`, new tags. Consumers of an earlier minor version within the same major version can safely ignore unknown fields.
- **PATCH** version increments indicate clarifications or documentation-only changes with no structural impact.

**Compatibility contract:**

- Within a major version, all changes are **additive only**. No existing field is removed or redefined.
- Consumers must ignore unknown fields gracefully (forward compatibility).
- Producers must populate all fields defined in their schema version; optional fields use `null` when not applicable.
- The initial schema version for Phase 1 is `"1.0.0"`. Phase 2 and Phase 3 additions increment the minor version (e.g., `"1.1.0"`, `"1.2.0"`).
- If a future phase requires breaking changes (e.g., restructuring the element hierarchy), the major version increments to `"2.0.0"`.

The `schema_version` field in the file-level manifest (Section 3.1) enables consumers to detect version mismatches and handle them appropriately — either by requesting regeneration or by applying version-specific parsing logic.

## 4. Addressing Scheme

### 4.1 Fully-Qualified Names (FQN)

Every manifest element is addressable by its FQN, which follows Python's own naming:

```
{module_path}.{class_name}.{method_name}
```

Examples:
- `startd8.contractors.artisan_contractor.ArtisanContractorWorkflow` — a class
- `startd8.contractors.artisan_contractor.ArtisanContractorWorkflow.run_phase` — a method
- `startd8.contractors.artisan_contractor._compute_gen_file_hash` — a module-level function
- `startd8.contractors.artisan_contractor._BUDGET_WARNING_THRESHOLD_FRACTION` — a constant

#### 4.1.1 FQN Edge Cases

**Elements inside conditional blocks:**

Functions, classes, or variables defined inside conditional blocks (`if __name__ == "__main__":`, `if TYPE_CHECKING:`, `if sys.platform == "win32":`, `try/except` bodies, etc.) are included in the manifest with the following rules:

- **`if __name__ == "__main__":` blocks**: Elements defined within this guard are included with their normal FQN (e.g., `module.main`, `module.setup_logging`). They receive an additional field `scope_guard: "__main__"` to indicate they are conditionally defined. These elements are typically not part of the module's importable API.

- **`if TYPE_CHECKING:` blocks**: Elements (typically imports and type aliases) within this guard are included with their normal FQN and `scope_guard: "TYPE_CHECKING"`. Imports within this block are also marked `is_conditional: true`.

- **`try/except` bodies**: Elements defined in `try` or `except` blocks are included with their normal FQN and `scope_guard: "try"` or `scope_guard: "except"` respectively.

- **Other conditional blocks** (`if`/`elif`/`else`, `for`, `while`): Elements are included with their normal FQN and `scope_guard` set to a descriptive string of the condition (e.g., `"if sys.platform == 'win32'"`, truncated to 80 characters).

**Duplicate names across conditional branches:**

When the same name is defined in multiple branches of a conditional (e.g., a function `impl` defined in both `if` and `else` blocks), each definition is included as a separate element. Disambiguation is achieved by appending `@branch[N]` to the FQN, where `N` is a zero-based index in source order:

- `module.impl@branch[0]` — definition in the `if` block
- `module.impl@branch[1]` — definition in the `else` block

If only one definition exists for a name (the common case), no `@branch` suffix is added.

**Nested scopes:**

For elements defined inside a function body (e.g., a helper function defined inside another function), the FQN follows Python's nesting: `module.outer_function.inner_function`. This applies regardless of conditional context — the `scope_guard` field captures the conditional context independently of the FQN path.

### 4.2 Span-Based Addressing

Every element also has a line-range span `[start_line, start_col, end_line, end_col]` that enables:

- **Extraction**: Read exactly the source code for one element
- **Replacement**: Replace an element's source while preserving surrounding code
- **Diff scoping**: Determine which elements a diff touches

**Comment and formatting preservation during surgical edits:**

Span-based addressing operates on the original source text, not on AST-reconstructed output. When replacing an element:

1. The consumer reads the original file content.
2. The text between `[start_line, start_col]` and `[end_line, end_col]` is replaced with the new source text.
3. All content outside the span — including comments, blank lines, and formatting — is preserved verbatim.
4. After replacement, the manifest for the modified file must be regenerated, as all spans below the edit point may have shifted.

This approach avoids the comment-loss problem inherent in `ast.unparse()` round-tripping. For complex structural edits (e.g., reordering methods, changing indentation levels), implementations should consider CST-based tooling (such as `libcst`) that can parse and reproduce comments and whitespace faithfully.

### 4.3 Addressing in Pipeline Prompts

Downstream consumers reference elements by FQN in structured instructions:

```
Modify element: startd8.contractors.artisan_contractor.ArtisanContractorWorkflow.run_phase
Located at: lines 120-185
Current signature: (self, phase: WorkflowPhase, context: dict) -> PhaseResult
Requested change: Add optional `timeout: float = None` parameter
```

This replaces the current pattern of regenerating entire files.

## 5. Generation Requirements

### 5.1 Input

- One or more Python source files (`.py`)
- Project root path (for FQN computation from file path)
- Optional: layer selection flags (AST-only, +symtable, +inspect, +dis)

When a file cannot be parsed due to syntax errors, encoding issues, or I/O failures, the generator follows the error handling strategy defined in Section 5.7.

### 5.2 Output Formats

| Format | Use Case |
|--------|----------|
| JSON | Machine consumption (pipeline stages, capability index) |
| YAML | Human review and configuration |
| Python dict | In-memory consumption within SDK |

### 5.3 Generation Modes

| Mode | Layers | Requires Import | Use Case |
|------|--------|-----------------|----------|
| `static` | AST + symtable | No | Default. Safe for any file. |
| `introspect` | AST + symtable + inspect | Yes | API surface mapping with resolved types. |
| `full` | All layers | Yes | Complete analysis including call graph. |

### 5.4 Determinism

- **Static mode**: Fully deterministic. Same source file always produces the same manifest.
- **Introspect/full modes**: Deterministic for a given Python version and environment, but import side effects may vary.

### 5.5 Performance

- Static mode must process a single file in < 100ms
- Batch processing (entire `src/startd8/` tree) must complete in < 10s
- Manifests are cached by content digest; regeneration only on source change

### 5.6 Staleness Detection

- Each manifest carries a `digest` field (SHA-256 of source content)
- Consumer code compares stored digest against current file hash
- Stale manifests trigger regeneration before use
- CLI flag: `--check` mode for CI integration (exit non-zero if manifests are stale)

### 5.7 Error Handling

Batch processing across a large codebase will inevitably encounter files that cannot be fully parsed. The manifest generator adopts a **graceful degradation** strategy:

**Syntax errors** (`SyntaxError` from `ast.parse()`):
- The generator emits a **partial manifest** for the file. The `elements` list is empty (or contains elements extracted before the error point, if a recovery parser is available). The `errors` field contains a `ParseError` entry with `kind: "syntax_error"`, the error message, and the line/column where parsing failed.
- Batch processing continues to the next file; a syntax error in one file does not halt generation for the entire project.

**Encoding errors** (files not decodable as UTF-8 or their declared encoding):
- The generator emits a manifest with empty `elements` and `imports`, and an `errors` entry with `kind: "encoding_error"`.
- The `digest` field is still computed from the raw bytes of the file.

**I/O errors** (permission denied, file not found, symlink loops):
- The generator emits a manifest with `kind: "io_error"` in `errors`. All structural fields are empty.

**Incomplete or truncated files:**
- Treated as syntax errors. If the file ends mid-expression, `ast.parse()` will raise `SyntaxError`, handled as above.

**Behavior flags:**
- `--strict` mode (opt-in): treat any parse error as a hard failure; exit non-zero immediately. Useful for CI when all files must be parseable.
- Default mode: collect all errors, emit partial manifests, report a summary at the end with counts of successful and failed files.

**Downstream consumer contract:** Consumers must check the `errors` field before relying on manifest content. A manifest with non-empty `errors` should be treated as incomplete — element lookups by FQN may return no results for elements that exist in the source but could not be parsed.

## 6. Pipeline Integration Points

### 6.1 Capability Index Build

**Current state:** Capabilities are declared in YAML manifests manually.

**With code manifest:**
- The build command scans source files and generates manifests
- Declared capabilities are validated against actual source structure
- Drift detection: "capability X references function Y which no longer exists"

## 8. Non-Requirements (Explicit Exclusions)

- **Code generation**: The manifest describes existing code; it does not generate new code.
- **Runtime execution**: Static mode (P0) must not import or execute the target file.
- **Language support beyond Python**: Future scope. This document covers Python only.
- **IDE integration**: The manifest format should be IDE-consumable but IDE plugins are out of scope.
- **Version control integration**: Git-aware diffing is a future enhancement, not a launch requirement.
- **Dynamically generated code in static mode**: Code produced at runtime via `exec()`, `eval()`, `type()` as a class factory, `collections.namedtuple()`, `dataclasses.make_dataclass()`, metaclass-generated methods, and similar dynamic patterns is **not represented** in static-mode manifests. The assignment statement (if present) appears as a `variable` element, but the synthesized attributes, methods, or class members are not discoverable without runtime introspection. Introspect mode (P2) may partially recover these. See also Section 2.1 "Limitations — Dynamically Generated Code".

### 11.3 Key Implementation Patterns

- **Frozen Pydantic models**: All data models use `ConfigDict(frozen=True)`. Mutations use `model_copy(update={...})` to produce new instances.
- **Parent annotation**: `_annotate_parents(tree)` attaches `_parent` refs to every AST node before visiting, enabling `_get_scope_guard()` to walk up the tree.
- **Pre-scan + visit**: Branch disambiguation uses a two-pass approach — `_scan_branch_names()` runs before the visitor, passing results via constructor param.
- **`model_dump(mode="json")`**: Used in `to_yaml()` to ensure enum values serialize as plain strings rather than Python-specific YAML tags.
