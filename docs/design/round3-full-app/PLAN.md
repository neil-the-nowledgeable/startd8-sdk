## Approach

Round 3 is **not a new scoring system** — it is a **fleet orchestrator** that composes shipped primitives:
`_run_checkout_cell` already proves the core move (bring up a sandboxed SUT, inject `*_SERVICE_ADDR`, run a gRPC suite, guaranteed teardown). Round 3 = **N of those at once, wired to each other, driven by a journey instead of a single-RPC suite.**

The **containment spike (M1) is DONE — verdict B**: the existing Seatbelt sandbox co-runs N untrusted servers (measured co-existence, loopback peer reachability, sustained external-egress denial, clean teardown), so **there is no Docker layer in v1**. The load-bearing build is therefore an **N-server GENERALIZATION of the existing `_run_checkout_cell` + `run_service_sandboxed`**: bind 9 servers on free randomized ports, inject each peer's `127.0.0.1:<port>` via the existing post-scrub `extra_env` path, readiness-gate dependency-ordered bring-up, and guarantee whole-fleet teardown on every exit path. NO netns/Docker rebuild. Build bottom-up: prove a **2-service** mesh (catalog ← recommendation, or cart ← checkout-with-one-real-dep) against an SDK reference build first; only then scale to 9 and spend LLM budget (reuse, don't regenerate — OQ-3). Topology = **single-model fleet** (OQ-1) and scoring = **layered** (OQ-2): both resolved (REQUIREMENTS §0).

**Hard prerequisite:** the in-flight **Track-2 expansion** must land the `startup` contracts for cart/catalog/recommendation/email (only checkout has one today). The fleet bring-up depends on those contracts to resolve serve commands + readiness probes; if the expansion hasn't landed them, Round 3 must add them itself (scope to budget).

**Platform caveat (v1 = macOS):** the verdict-B substrate is macOS Seatbelt shared-loopback. On Linux CI the sandbox auto-selects `unshare -rn` (per-netns) which removes the FR-7a weakness but **breaks** shared-loopback peer reachability — a Linux fleet needs a shared-netns/veth design (REQUIREMENTS OQ-7). v1 does not target Linux; that is a follow-up.

## Sequenced steps

> **M1 is the containment feasibility gate and is DONE** (verdict B — see CONTAINMENT_SPIKE.md). M2 is the load-bearing N-server fleet generalization. **Prerequisite gate before M2:** Track-2 startup contracts for cart/catalog/recommendation/email must land (or be added here).

0. **M0 — topology + roster (no LLM, no launch).** `topology.py` parses `demo.proto` → the 9-node dep graph + `*_SERVICE_ADDR` map. `roster.py` resolves a finalist's 9 workdirs from durable batch roots by verbatim model id. *Test:* graph matches the checkout seed's declared 6 deps; missing service → `service-missing:S`.
1. **M1 — containment spike (feasibility gate) — ✅ DONE, verdict B.** Drove the real `run_service_sandboxed` co-running N untrusted servers. **MEASURED:** N co-exist without serialization (3 concurrent, 0.87s); sandboxed A reaches sandboxed B over loopback; external egress stays DENIED per-process; clean process-group teardown, zero orphans. **Outcome: the existing sandbox suffices → no Docker layer in v1.** Recorded weakening (shared unrestricted loopback plane → any service can dial any sibling/host loopback port) accepted as **FR-7a** with mitigations + a Docker escalation trigger. See CONTAINMENT_SPIKE.md.
2. **M2 — N-server fleet generalization (the load-bearing step).** `fleet.py` `FleetHarness.start()/stop()` generalizes `_run_checkout_cell` + `run_service_sandboxed` to N: bind each service on a free **randomized** port, inject each peer `*_SERVICE_ADDR` via the existing post-scrub `extra_env`, **dependency-ordered** readiness-gated bring-up (reuse the checkout B4 ordering for the TOCTOU window that now scales with N), guaranteed **whole-fleet** teardown on every exit path. Prove first on a **2-service** mesh against SDK reference builds (catalog←recommendation), then scale. *Test:* SDK reference builds → journey step passes; known-broken upstream → downstream step classified `propagated`, not `model-fault`; all servers reaped on every path. **NO Docker.**
3. **M3 — journeys + fleet oracle.** `journeys.py` canonical browse→cart→checkout→confirm; `oracle.py` promotes the GroundTruth math. Run the journey against a **full SDK reference 9-service mesh** (all reference builds) → 100% step coverage. This is the R2-S3 discipline: the journey must be proven on a known-good mesh before any model is scored.
4. **M4 — full 9-service fleet + layered attribution scoring.** `score.py` per-step coverage (headline) + per-service fault classes (FR-14) + derived "canonical journey completed" boolean (FR-13 layered, OQ-2). Run on the SDK reference mesh + 1–2 deliberately-broken meshes (break payment; break catalog) → faults attributed to the right service with the right class.
5. **M5 — real finalists (single-model fleets; reuse workdirs, minimal/no new LLM spend per OQ-3) + report + CLI + decision.** Each finalist's own 9-service fleet deployed + scored as that model's system (OQ-1). Persist per-cell (Mottainai), render `round3-system-report.{json,md}`, finalist leaderboard. Decision gate: does the journey discriminate finalists, and is attribution trustworthy?

## Discoveries table (draft assumption → what the code/infra revealed)

| Draft assumption (v0.1) | What planning revealed in the code/infra |
|---|---|
| The deploy harness is the Round-3 foundation; it can bring up the fleet. | **Partly.** `deploy_app_local` is **single-app, single-venv, single-port** by construction (one `app_root`, one `LiveServer`, one entrypoint). `deploy_batch` deploys roots **independently and serially** ("avoids ephemeral port races"), explicitly *not* together. Its `context_smoke` dials *pre-existing remote* producers by `base_url` — it never *stands up* a producer fleet. So deploy_harness gives us **discovery + venv/install (Python) + readiness/health patterns to reuse**, but the **co-deploy + inter-wiring is net-new.** |
| The checkout-stub pattern can scale to deploying all 9 real services wired together. | **The pattern, yes; the code, no.** `_run_checkout_cell` is the *exact* primitive — bring up 1 sandboxed SUT, inject 6 `*_SERVICE_ADDR`, run a suite, `finally` tear down — but its 6 deps are **SDK stubs** (`CheckoutStubHarness`, in-process Python gRPC servers), launched **in-process** in the harness, while only the *one* SUT is sandboxed. Round 3 needs 9 **sandboxed generated** servers co-running and wired to each other. So `_run_checkout_cell` is the **blueprint to generalize**, not reusable as-is. |
| The sandbox supports multi-service co-launch. | **Net-new orchestration, but the substrate WORKS (spike verdict B, MEASURED).** `run_service_sandboxed` runs **one** untrusted server (`os.setsid()` own process group, one readiness wait, one client window, one guaranteed teardown). The spike drove it **N-concurrent**: servers co-exist without serialization (3 in 0.87s), a sandboxed peer is reachable over loopback (the wiring we want), external egress stays DENIED, teardown reaps every process with zero orphans. So **no Docker is needed in v1** — the work is the N-server *orchestration* (`fleet.py`), not a new isolation substrate. Recorded weakening: the loopback plane is **shared and unrestricted** (any service can dial any sibling/host loopback port) — accepted as FR-7a (same-model fleet, disposable host) with a Docker escalation trigger. Linux uses a different substrate (per-netns) → OQ-7. |
| There is a frontend / loadgenerator to drive the journey. | **There is not.** `seeds/` has the **9 backend services only** (no `frontend`, no `loadgenerator`). The "user journey" driver **must be SDK-authored** (a gRPC client sequence, reusing the `checkout_suite` client). This is actually cleaner — the SDK must own the oracle anyway — and resolves OQ-4. |
| Round 3 needs to generate 9 services per finalist. | **No — reuse them (Mottainai).** Per-service generated workdirs already exist in durable batch roots with `.model` sidecars (`deploy_harness.batch`). Round 3 should **reuse** them, regenerating only gaps (OQ-3). Open: verify Round-1/2 services are mesh-compatible (same `demo.proto`, same addr convention) — they *should* be, since the checkout seed already encodes the exact `*_SERVICE_ADDR` convention, but unconfirmed for the non-checkout services' env contracts. |
| NR-X2 forbids generated deps, so Round 3 violates the parent design. | **It crosses the frontier the parent *named* (NR-X1 live-mesh + NR-X2 generated-deps), but under the recommended single-model topology it honors NR-X2's *intent*.** NR-X2 forbids a service dialing **another model's** generated dep (confounds *across* models). Single-model-fleet keeps every dep the *same* model's code → the confound is *within one model*, which is the legitimate "can this model build a system" question. The expansion plan's own line ("if checkout's deps were *generated*… checkout's score would absorb its deps' quality") is exactly why **best-of-breed (OQ-1 b) is the wrong default.** |
| Scoring is a new system. | **No.** Reuse `aggregate.py` (median/IQR/leaderboard), the `BehavioralResult` model-fault-vs-degrade discipline, durable persistence, and the markdown rollup. Round 3 adds **journey step coverage + per-step fault attribution** on top, not a new scorer. |
| Each service has a startup contract. | **Only checkout does today** (`startup.dependency_addr_env`); cart/catalog/recommendation/email startup blocks are being added by the in-flight Track-2 expansion (E2). Round 3 **depends on that expansion landing** (or must add the startup blocks itself) — a real sequencing dependency. |

## Risks / feasibility blockers

- **Containment (OQ-6/FR-7a) — RESOLVED by the M1 spike (verdict B), no longer the #1 blocker.** The shipped sandbox co-runs N untrusted servers (measured); no Docker in v1. Residual: the **shared unrestricted loopback plane** weakening is *accepted* (FR-7a) with mitigations + a Docker escalation trigger — not a blocker for same-model fleets on a disposable host. The operational part survives below (TOCTOU scales with N).
- **#1 remaining risk (post-spike) — sequencing dependency on the Track-2 expansion:** the fleet bring-up needs `startup` contracts (serve command + readiness probe) for all 9 services, but **only checkout has one today** — cart/catalog/recommendation/email are in the in-flight Track-2 expansion. M2 **cannot start** until those land (or Round 3 adds them itself = scope creep to budget). This is now the gating dependency on the critical path.
- **Operational: port-ordering / readiness TOCTOU scales with N** — the `_free_port`→bind window is per-server; with 9 the collision/race chance rises and partial-teardown leaks are costlier. Reuse the checkout **B4 ordering** discipline and the existing readiness-poll degrade so a slow/failed peer degrades the cell rather than misscoring.
- **Mesh-compatibility of reused workdirs (OQ-3):** unverified that Round-1/2 non-checkout services honor the `*_SERVICE_ADDR` env convention at runtime (they were generated in isolation). If they don't, the "reuse" saving partly evaporates.
- **Attribution correctness (FR-14):** distinguishing `model-fault:S` from `propagated:S←D` across a 6-deep fan-out is subtle; a mis-attribution silently blames the wrong model. The known-broken-mesh tests (M4) are the gate.
- **Linux substrate (OQ-7, open):** v1 is macOS-only; a Linux fleet needs a shared-netns/veth design (per-netns `unshare -rn` breaks peer reachability). Deferred follow-up, not a v1 blocker.

## Reuse vs net-new (the headline)

**Reusable (~50–60%):** discovery/venv/install (Python), `run_service_sandboxed` (per-service), `resolve_serve_command`/`StartupContract`, the `*_SERVICE_ADDR` injection convention, `CheckoutStubHarness` GroundTruth math, the `checkout_suite` client pattern, `aggregate.py` + leaderboard + durable persistence + verbatim-id join.
**Net-new (~40–50%):** the **fleet orchestrator** (`fleet.py` — ordered co-launch + inter-wiring + whole-fleet teardown), the **topology graph** (`topology.py`), the **journey definitions + fleet oracle** (`journeys.py`/`oracle.py`), the **step-coverage + cross-mesh attribution scorer** (`score.py`), the **system report + CLI** (`report.py`).

## Size estimate

**Larger than Round 2's per-service expansion.** Track-2 expansion = "suites + state fixtures + a dependency-stub harness + seed startup contracts, **not new scoring**." Round 3 adds a **net-new orchestration layer** (fleet co-deploy, topology, multi-process teardown) and a **net-new scoring dimension** (layered: journey step coverage + cross-mesh attribution + derived boolean). **The spike removed the largest unknown:** no Docker/v2 isolation upgrade in v1 (verdict B), so the size does **not** grow by an isolation substrate. Rough order: **~1.5–2× the expansion milestone**, now dominated by `fleet.py` (the N-server generalization) and the layered scorer — and **gated on the Track-2 startup contracts landing first** (now the critical-path dependency).

## Out of scope (this plan)
Auto-finalist-selection (NR-1/NR-7); generated frontend/loadgenerator (NR-2/OQ-4); load/perf/chaos (NR-3); best-of-breed as the default topology (OQ-1 RESOLVED → single-model fleet; best-of-breed = optional diagnostic only); real external persistence backends (NR-5); **a Docker/netns isolation substrate (OQ-6 verdict B — not needed in v1)**; **a Linux fleet substrate (OQ-7 — deferred follow-up)**.
