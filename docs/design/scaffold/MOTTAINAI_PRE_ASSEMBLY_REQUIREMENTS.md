## 1. Goal

Eliminate redundant LLM work by performing deterministic assembly, element registry population, and trivial element filling **at plan ingestion time**, then forwarding the pre-assembled artifacts through SCAFFOLD and IMPLEMENT so that cloud LLM prompts are scoped to only the non-trivial method bodies that require stochastic generation.

This is a Mottainai compliance initiative. The infrastructure already exists — `DeterministicFileAssembler`, the template registry, `ElementRegistry`, `make_element_id()`, and the complexity classifier — but the pipeline does not wire them together at the earliest possible moment. The result is four classes of waste:

| # | Waste | Where | Impact |
|---|---|---|---|
| W-1 | Skeleton source computed at ingestion, discarded, recomputed at SCAFFOLD | `plan_ingestion_workflow.py` line 3556 → `scaffold.py` line 182 | Duplicate CPU computation |
| W-2 | Template-matchable elements (`__init__`, `__repr__`, constants) generated by cloud LLM | IMPLEMENT phase, every run | ~15-25% of cloud output tokens |
| W-3 | Cloud prompts describe entire files when only method bodies are needed | IMPLEMENT spec/drafter prompts | ~30-50% of cloud input tokens |
| W-4 | Elements generated in run N re-generated from scratch in run N+1 | Cross-run: no registry lookup at ingestion | Full cloud cost repeated per element |

## 2. Problem Statement

### 2.1 Current Pipeline

```
Plan Ingestion EMIT
  ├── extract_forward_contracts() → ForwardManifest ✓
  ├── DeterministicFileAssembler.render_specs() → RenderResult ✓
  ├── Store render_result.metadata in seed (stub_manifest) ✓
  ├── DISCARD render_result.specs (the actual source text) ← W-1
  ├── ElementRegistry exists but NOT populated ← W-4
  ├── make_element_id() runs in extractor but IDs not indexed ← W-4
  └── No per-element classification or template matching ← W-2

SCAFFOLD Phase
  ├── Re-extract ForwardManifest from seed ✓
  ├── Re-instantiate DeterministicFileAssembler ← W-1
  ├── Re-run render_specs() (duplicate computation) ← W-1
  ├── Materialize skeletons to disk ✓
  └── Skeletons contain raise NotImplementedError stubs ✓

IMPLEMENT Phase
  ├── Cloud LLM receives: "Generate rate_limiter.py" ← W-3
  ├── Cloud LLM generates: class skeleton + imports + __init__ + logic ← W-2
  └── Template-matchable elements re-derived stochastically ← W-2
```

### 2.2 Infrastructure Already Built But Not Wired

| Component | Module | Status | Used at Ingestion? |
|---|---|---|---|
| `DeterministicFileAssembler` | `utils/file_assembler.py` | Implemented (724 lines) | Partially — renders but discards source |
| `TemplateRegistry` | `micro_prime/templates.py` | Implemented (731 lines) | No |
| `ElementRegistry` | `element_registry.py` | Implemented (670 lines) | No |
| `make_element_id()` | `element_id.py` | Implemented (312 lines) | Yes — IDs generated but not indexed |
| `classify_tier()` | `complexity/classifier.py` | Implemented | No |
| `extract_signals_from_element()` | `complexity/signals.py` | Implemented | No |
| `ForwardManifest.get_element_by_id()` | `forward_manifest.py` | Implemented | No — lazy index never triggered |

Seven implemented modules, zero wired at ingestion. This is the core Mottainai violation: the capability exists, but the pipeline discards or defers what it could use immediately.

### 2.3 Key Observation

By the time plan ingestion EMIT runs, the pipeline has everything needed for deterministic pre-assembly:

| Data | Source | Sufficient For |
|---|---|---|
| Element names, kinds, signatures | `ForwardManifest.file_specs[].elements[]` | Full skeleton generation |
| Deterministic element IDs | `make_element_id()` in `forward_manifest_extractor.py` | Registry population + cross-run lookup |
| Import statements | `ForwardManifest.file_specs[].imports[]` | Import block assembly |
| Base classes, decorators | `ForwardElementSpec.bases`, `.decorators` | Class hierarchy rendering |
| Parameter lists with types | `ForwardElementSpec.signature.params[]` | `__init__` template matching |
| Interface contracts | `ForwardManifest.contracts[]` | Contract-aware stub bodies + context checksums |
| Complexity signals | `api_signatures`, `estimated_loc`, `protocol` | Per-element tier classification |
| Prior generated code | `ElementRegistry` at `.startd8/state/elements/` | Cross-run element reuse |

### 2.4 Proposed Pipeline

```
Plan Ingestion EMIT
  ├── extract_forward_contracts() → ForwardManifest
  ├── Build manifest element index (O(1) lookup)                    [FR-MPA-009]
  ├── Populate ElementRegistry with all specs                       [FR-MPA-009]
  ├── Compute context checksums per element                         [FR-MPA-011]
  ├── Cross-run registry lookup: reuse prior generated code         [FR-MPA-010]
  ├── classify_tier() per element → per-element tier map            [FR-MPA-002]
  ├── DeterministicFileAssembler.render_specs() → RenderResult
  ├── TemplateRegistry.match() per TRIVIAL element → fill bodies    [FR-MPA-003]
  ├── Registry pre-fill: inject cached code into stubs              [FR-MPA-010]
  ├── Store render_result.specs in seed (skeleton_sources)          [FR-MPA-001]
  ├── Store per-element tier + fill status in seed                  [FR-MPA-002]
  └── Store stub_manifest metadata (existing)

SCAFFOLD Phase
  ├── Read skeleton_sources from seed                               [FR-MPA-004]
  ├── Materialize to disk (skip re-render unless module_inventory
  │   changes import classification)                                [FR-MPA-004]
  └── Drift detection: compare seed SHA vs materialized SHA         [existing]

IMPLEMENT Phase
  ├── For tasks with pre-assembled skeletons:
  │   ├── Prompt: "Fill method bodies in existing skeleton"         [FR-MPA-005]
  │   ├── Context: skeleton source with trivial bodies pre-filled   [FR-MPA-005]
  │   └── Scope: only unfilled elements as implementation targets   [FR-MPA-005]
  ├── For tasks without skeletons: current behavior (unchanged)
  └── Register generated elements in ElementRegistry                [existing ER-007]
```

## 3. Scope

### 3.1 In Scope

**Pre-assembly (new):**
- Persisting rendered skeleton source text in the ingestion seed
- Per-element complexity classification at ingestion time
- Template registry pre-fill for trivial elements during ingestion
- SCAFFOLD consumption of pre-rendered skeletons (avoiding recomputation)
- IMPLEMENT prompt narrowing for files with pre-assembled skeletons
- Edit-mode prompt injection for all tasks with existing skeleton files
- OTel metrics for pre-assembly coverage and cost avoidance

**Element Registry at ingestion (subsumed from ER-003, REQ-MP-1101, 1104, 1105, 1106, 1108):**
- Populate `ElementRegistry` with all ForwardElementSpecs during EMIT
- Build manifest element index for O(1) lookup by ID
- Compute per-element context checksums for staleness detection
- Cross-run element lookup: check registry for code from prior runs
- Registry-based skeleton pre-fill: inject cached code instead of `raise NotImplementedError`
- Skeleton-derived element specs for features without `api_signatures`

### 3.2 Out of Scope

- Ollama-based generation at ingestion time (requires Ollama availability; deferred)
- Changes to plan ingestion LLM prompts (covered by DFA Requirements §8)
- Element Registry `put()` of newly generated code (remains at IMPLEMENT time — code doesn't exist until generation; covered by ER-007)
- Phase-specific status tracking in registry (ER-006 through ER-011 — each depends on its phase executing)
- Cross-run Kaizen metrics (ER-014) and Warm Up reconciliation (ER-015)
- Modifications to the Micro Prime engine internals
- Non-Python file types

### 3.3 Boundary: What Happens at Ingestion vs. Generation

| Concern | Ingestion (this doc) | Generation (ER-007, ER-012) |
|---|---|---|
| **Element inventory** | `put(ElementEntry(code=None, status="specified"))` | — |
| **Context checksums** | Computed from spec + contracts | Validated at lookup time |
| **Cross-run reuse** | `get()` → attach cached code if checksum matches | `get()` → skip generation if code cached |
| **Template fill** | `TemplateRegistry.match()` for TRIVIAL elements | Same (Micro Prime already does this) |
| **Code registration** | — | `put(ElementEntry(code=generated_code))` |
| **Phase status** | `set_phase_status("plan_ingestion", "specified")` | `set_phase_status("implement", "generated")` |

The principle: **establish the inventory and reuse what's available at the earliest moment; register new code at the moment it's produced.**

## 4. Functional Requirements

### FR-MPA-001: Persist Skeleton Source Text in Seed

At plan ingestion EMIT, after `DeterministicFileAssembler.render_specs()` succeeds, the pipeline SHALL store `render_result.specs` (the `Dict[str, str]` mapping file paths to validated Python source text) in the seed under `artifacts["skeleton_sources"]`.

**Rationale:** The source text is already computed and AST-validated. Discarding it forces SCAFFOLD to recompute identically. This is Mottainai Rule 2 ("forward, don't regenerate").

**Size budget:** Typical skeleton source is 30–80 lines per file. For a 20-file plan, this adds ~40–80 KB to the seed JSON. Acceptable given seeds already contain full design documents (often 200+ KB).

**Schema:**
```python
# In seed["artifacts"]
"skeleton_sources": {
    "src/rate_limiter/limiter.py": "# [STARTD8-SKELETON]\n\nfrom __future__...",
    "src/rate_limiter/config.py": "# [STARTD8-SKELETON]\n\nfrom __future__...",
}
```

### FR-MPA-002: Per-Element Complexity Classification at Ingestion

At plan ingestion EMIT, after ForwardManifest extraction, the pipeline SHALL classify each `ForwardElementSpec` into a complexity tier (TRIVIAL, SIMPLE, MODERATE, COMPLEX) using the existing `classify_tier()` function from `complexity/classifier.py`.

The per-element tier map SHALL be stored in the seed under `artifacts["element_tiers"]`.

**Schema:**
```python
# In seed["artifacts"]
"element_tiers": {
    "src/rate_limiter/limiter.py": {
        "RateLimiter": {"tier": "MODERATE", "element_id": "class/src.rate_limiter.limiter.ratelimiter-a1b2c3d4e5f6"},
        "RateLimiter.__init__": {"tier": "TRIVIAL", "element_id": "method/..."},
        "RateLimiter.check_quota": {"tier": "SIMPLE", "element_id": "method/..."},
        "RateLimiter.reset_window": {"tier": "SIMPLE", "element_id": "method/..."},
        "RateLimiter.__repr__": {"tier": "TRIVIAL", "element_id": "method/..."},
        "DEFAULT_WINDOW_SEC": {"tier": "TRIVIAL", "element_id": "constant/..."},
    }
}
```

**Signal extraction:** Use `extract_signals_from_element()` from `complexity/signals.py`, which accepts a `ForwardElementSpec` and its parent `ForwardFileSpec`. Signals available at ingestion time:

| Signal | Available | Source |
|---|---|---|
| `estimated_loc` | Yes (from element signature complexity) | Heuristic: params × 3 + bases × 5 |
| `blast_radius` | Partial (0 for create-mode files) | ForwardManifest cross-references |
| `caller_count` | No (requires existing codebase) | Default to 0 for new files |
| `has_dynamic_dispatch` | Yes (from decorators list) | `ForwardElementSpec.decorators` |
| `edit_mode` | Yes ("create" for new files) | Inferred from seed |
| `manifest_coverage` | Yes ("full" — element is in manifest) | Always "full" at this stage |
| `target_file_count` | Yes (1 — element-level) | Always 1 |
| `mro_depth` | Partial (from bases list length) | `len(ForwardElementSpec.bases)` |

**Constraint:** Classification at ingestion is advisory. Downstream phases MAY reclassify elements when additional signals (existing codebase, caller graph) become available. The ingestion-time classification optimizes the common case (create-mode, new files) where most elements are TRIVIAL or SIMPLE.

### FR-MPA-003: Template Registry Pre-Fill at Ingestion

At plan ingestion EMIT, after per-element classification (FR-MPA-002), the pipeline SHALL attempt template matching for each TRIVIAL-classified element using the existing `TemplateRegistry` from `micro_prime/templates.py`.

For each element where `TemplateRegistry.match()` returns a non-None result:

1. The template-rendered body SHALL replace the `raise NotImplementedError` stub in the skeleton source text
2. The element SHALL be marked as `"pre_filled": true` in the element tier map
3. The filled source SHALL be re-validated via `ast.parse()` after template insertion

**Schema extension:**
```python
# In seed["artifacts"]["element_tiers"][file_path]
"RateLimiter.__init__": {
    "tier": "TRIVIAL",
    "element_id": "method/...",
    "pre_filled": true,
    "fill_source": "template:init_with_defaults"
}
```

**Failure handling:** If template rendering produces invalid Python (fails `ast.parse()`), the element SHALL revert to the `raise NotImplementedError` stub. Template failures are logged but do not block ingestion.

**Templates applicable at ingestion time:**

| Template | Match Condition | Ingestion Feasibility |
|---|---|---|
| `init_store_params` | `__init__` with all positional params | Yes — params in signature |
| `init_with_defaults` | `__init__` with optional params | Yes — defaults in signature |
| `init_varargs` | `__init__` with `*args`/`**kwargs` | Yes — param kinds in signature |
| `repr_from_init` | `__repr__` method | Yes — reads sibling `__init__` params |
| `eq_by_attrs` | `__eq__` method | Yes — reads sibling `__init__` params |
| `hash_by_attrs` | `__hash__` method | Yes — reads sibling `__init__` params |
| `constant_value` | `CONSTANT` with `value_repr` | Yes — value in element spec |
| `type_alias` | `TYPE_ALIAS` kind | Yes — annotation in signature |
| `simple_validation` | Single-param validation | Partial — needs heuristic |
| `dataclass_boilerplate` | `@dataclass` in decorators | Yes — fields from params |

### FR-MPA-004: SCAFFOLD Consumes Pre-Rendered Skeletons

The SCAFFOLD phase SHALL check for `artifacts["skeleton_sources"]` in the seed context before invoking `DeterministicFileAssembler.render_specs()`.

When `skeleton_sources` is present and non-empty:

1. Use the pre-rendered source text directly for materialization
2. Skip the `render_specs()` call (avoid duplicate computation)
3. Run `_check_stub_drift()` comparing seed SHA256 (from `stub_manifest`) against materialized content
4. If `module_inventory` is available and non-empty, optionally re-classify imports in the pre-rendered source (import tier upgrade). This is the only case where re-rendering is justified.

When `skeleton_sources` is absent or empty:

1. Fall back to current behavior: call `render_specs()` from ForwardManifest

**Rationale:** Eliminates W-1 (duplicate computation). The pre-rendered source is already AST-validated at ingestion time.

### FR-MPA-005: IMPLEMENT Edit-Mode Prompt for Pre-Assembled Files

When the IMPLEMENT phase processes a task whose target files have pre-assembled skeletons on disk (created by SCAFFOLD from ingestion-time assembly), the implementation prompt SHALL:

1. Include the skeleton source text as context (the file already exists)
2. List only the unfilled elements (`pre_filled: false` or no tier entry) as implementation targets
3. Use edit-mode framing: "Implement the bodies of the following methods in the existing file" rather than "Generate a complete Python module"
4. Exclude template-filled and registry-filled elements from the implementation scope

**Prompt structure:**
```
## Existing File: src/rate_limiter/limiter.py

The following skeleton file already exists with correct imports, class
structure, and some method bodies pre-filled. Implement ONLY the method
bodies marked with `raise NotImplementedError`.

### Pre-filled (do not modify):
- RateLimiter.__init__ (deterministic template)
- RateLimiter.__repr__ (deterministic template)
- DEFAULT_WINDOW_SEC (constant)
- RateLimiter._parse_config (registry: prior run)

### To implement:
- RateLimiter.check_quota(self, key: str) -> bool
- RateLimiter.reset_window(self, key: str) -> None
- RateLimiter._evict_expired(self) -> int

<skeleton>
{skeleton_source}
</skeleton>
```

**Token savings estimate:** For a typical file with 8 elements where 3 are template-filled and 1 is registry-filled:
- Input reduction: ~200–400 tokens (skeleton provided as context vs. described in prose)
- Output reduction: ~200–500 tokens (4 elements not generated by LLM)
- Net savings: ~20–40% per file

### FR-MPA-006: Artisan Inner Loop Prompt Narrowing

For the artisan contractor (8-phase pipeline), the `ImplementPhaseHandler` and `DevelopmentPhase` SHALL detect when skeleton files exist in the project root and adjust chunk-level prompts accordingly:

1. Read the skeleton file from disk (written by SCAFFOLD)
2. Identify which elements are already filled (no `raise NotImplementedError`)
3. Narrow the `implementation_prompt` to reference only unfilled elements
4. Set `edit_mode: true` in the chunk context so the LLM receives edit-mode instructions

**Integration point:** `LLMChunkExecutor` in `development.py` constructs per-chunk prompts. The prompt construction SHALL check for existing skeleton content and, when present, switch from "generate file" to "fill stubs" framing.

### FR-MPA-007: Prime Contractor Skeleton Forwarding

For the Prime Contractor path (`MicroPrimeCodeGenerator` in `prime_adapter.py`), the adapter already accepts a `skeletons` dict and falls back to auto-generation via `_generate_skeletons()` when missing.

The adapter SHALL preferentially consume `skeleton_sources` from the seed context before auto-generating:

1. Check `context.get("skeleton_sources")` (from ingestion)
2. If present, use as the `skeletons` parameter (skip `_generate_skeletons()`)
3. Template-filled and registry-filled bodies in the skeleton source are preserved — Micro Prime's splicer only replaces `raise NotImplementedError` stubs
4. Log the number of pre-filled elements consumed from ingestion

**Rationale:** Prime Contractor already generates skeletons on-demand (REQ-MP-702). This requirement upgrades the skeleton source from "auto-generated at generation time" to "pre-computed at ingestion time with trivial bodies and cached code filled."

| `mottainai.elements_pre_filled` | Counter | `fill_source` (template/registry), `phase=ingestion` | Elements filled without LLM |
| `mottainai.registry_hits` | Counter | `phase=ingestion` | Cross-run element reuse hits |
| `mottainai.registry_misses` | Counter | `phase=ingestion` | Cross-run element reuse misses |

### FR-MPA-009: Element Registry Population at Ingestion (subsumes ER-003, REQ-MP-1101, REQ-MP-1104)

At plan ingestion EMIT, after ForwardManifest extraction, the pipeline SHALL populate the `ElementRegistry` with all discovered elements.

**9a. Registry population (ER-003):**

For each `ForwardElementSpec` across all `ForwardManifest.file_specs`, create and store an `ElementEntry`:

```python
ElementEntry(
    element_id=spec.id,                    # from make_element_id() — already computed
    code=None,                             # no generated code yet
    kind=spec.kind.value,
    name=spec.name,
    file_path=file_spec.file,
    parent_class=spec.parent_class,
    source_contract_id=spec.id,
    context_checksum=_element_context_checksum(spec, contracts),
)
```

Initial phase status: `set_phase_status(element_id, "plan_ingestion", "specified")`.

**9b. Manifest element index (REQ-MP-1101):**

Trigger `ForwardManifest.get_element_by_id()` to build the lazy `_element_id_index` during EMIT. This ensures O(1) lookups are available for all downstream phases without redundant index construction.

**9c. Skeleton-derived element specs (REQ-MP-1104):**

For features where `api_signatures` is empty and the DeterministicExtractor produced zero `ForwardElementSpec` entries, but a skeleton file was rendered by DFA:

1. Parse the skeleton AST
2. Create `ForwardElementSpec` entries for each `def`/`class` containing `raise NotImplementedError`
3. Assign IDs with format `flcm-skel-{relpath}:{line}:{name}`
4. Add specs to the ForwardManifest (low precedence — below `deterministic`, `human-yaml`, `proto`, `source-ast`)
5. Register in `ElementRegistry` with the same flow as 9a

**Rationale:** Features without `api_signatures` currently produce zero element specs, causing Micro Prime to receive an empty elements list and fall through to blind cloud generation. Skeleton-derived specs close this gap deterministically.

**Constraints:**
- Registry I/O failures SHALL NOT abort ingestion — log warning and continue (Mottainai Rule 3: degrade gracefully)
- The `ElementRegistry` is initialized with `state_dir = output_dir / ".startd8" / "state"` (same location used by artisan resume cache)
- `schema_version` field included in all entries for forward compatibility

### FR-MPA-010: Cross-Run Element Lookup at Ingestion (subsumes REQ-MP-1105, REQ-MP-1106)

At plan ingestion EMIT, after registry population (FR-MPA-009), the pipeline SHALL check for existing generated code from prior runs.

**10a. Cross-run element lookup (REQ-MP-1105):**

For each `ForwardElementSpec`, call `element_registry.get(element_id)`:

- **HIT with valid context checksum** → element has code from a prior run that is structurally compatible with the current spec. Mark as `"fill_source": "registry:{source_task}"` in the element tier map.
- **HIT with stale context checksum** → element's signature or contracts changed since the cached code was generated. Invalidate the entry (`registry.remove(element_id)`), treat as MISS.
- **MISS** → no cached code. Element remains a `raise NotImplementedError` stub.

**10b. Registry-based skeleton pre-fill (REQ-MP-1106):**

For elements with a registry HIT (valid checksum), inject the cached `ElementEntry.code` into the skeleton source text, replacing the `raise NotImplementedError` stub:

1. Locate the stub in the skeleton source (match `def {name}(...):` followed by `raise NotImplementedError`)
2. Replace the stub body with the cached code
3. Re-validate via `ast.parse()` — if the cached code produces a syntax error in the new skeleton context (e.g., imports changed), revert to stub and log warning
4. Mark the element with a traceability comment: `# [ELEMENT-REGISTRY: {element_id}]`

**Decision tree per element at ingestion:**

```
ForwardElementSpec
  │
  ├── 1. Registry lookup: registry.get(element_id)
  │     ├── HIT + checksum valid → inject cached code → "fill_source: registry"
  │     ├── HIT + checksum stale → invalidate → fall through
  │     └── MISS → fall through
  │
  ├── 2. classify_tier() → TRIVIAL / SIMPLE / MODERATE / COMPLEX
  │
  ├── 3. If TRIVIAL and not already filled:
  │     └── TemplateRegistry.match() → fill body → "fill_source: template:{name}"
  │
  └── 4. Otherwise: stub remains → "fill_source: none"
```

**Precedence:** Registry code takes priority over template code. If an element has both a registry hit and a template match, use the registry code (it was generated by an LLM and reviewed — higher fidelity than a deterministic template).

**Cross-run value example (Run 004 waste from REQ-MP-11xx §1.6):**

| Feature Pair | Shared Element | Run 1 Cost | Run 2 Cost (current) | Run 2 Cost (with FR-MPA-010) |
|---|---|---|---|---|
| PI-001 / PI-002 | `getJSONLogger`, `add_fields` | $0.094 | $0.094 (re-generated) | $0.00 (registry hit) |
| PI-003 / PI-006 | `initStackdriverProfiling` | $0.187 | $0.187 (re-generated) | $0.00 (registry hit) |

### FR-MPA-011: Context Checksums at Ingestion (subsumes REQ-MP-1108)

At plan ingestion EMIT, the pipeline SHALL compute a context checksum for each `ForwardElementSpec` using the element's structural context:

```python
def _element_context_checksum(
    element: ForwardElementSpec,
    contracts: list[InterfaceContract],
) -> str:
    """Hash the element's structural context for staleness detection."""
    parts = [
        element.name,
        element.kind.value,
        str(element.signature) if element.signature else "",
        element.parent_class or "",
        ",".join(sorted(element.bases)),
        ",".join(sorted(element.decorators)),
        *(c.checksum for c in contracts
          if c.function_name == element.name
          or c.class_name == element.parent_class),
    ]
    return hashlib.sha256("|".join(parts).encode()).hexdigest()
```

The checksum SHALL be:

1. Stored in the `ElementEntry.context_checksum` field when populating the registry (FR-MPA-009)
2. Compared at cross-run lookup time (FR-MPA-010) to detect structural changes
3. Stored in the element tier map: `"context_checksum": "a1b2c3..."` for downstream staleness checks

**Invalidation triggers:**

| Trigger | Detection | Action |
|---|---|---|
| Signature changed (params, return type) | Checksum mismatch at lookup | Invalidate entry, treat as MISS |
| Interface contract changed | Contract checksum in hash input | Invalidate entry, treat as MISS |
| Base classes changed | Bases in hash input | Invalidate entry, treat as MISS |
| Decorators changed | Decorators in hash input | Invalidate entry, treat as MISS |
| Manual invalidation | `--clean-element-cache` CLI flag | Clear all entries |

## 5. Non-Functional Requirements

### NFR-MPA-001: Zero LLM Cost at Ingestion

All pre-assembly work (skeleton rendering, element classification, template matching, registry lookup) SHALL make zero LLM API calls. This is deterministic computation and disk I/O only.

### NFR-MPA-002: Seed Size Budget

The addition of `skeleton_sources` and `element_tiers` to the seed SHALL not increase seed JSON size by more than 150 KB for plans with up to 30 files. If a plan produces skeleton sources exceeding this budget, the pipeline SHALL store only files whose elements include at least one TRIVIAL-tier or registry-hit element (i.e., skip files where pre-assembly adds no value).

### NFR-MPA-003: Backward Compatibility

Seeds produced without `skeleton_sources`, `element_tiers`, or registry data SHALL continue to work. All consuming code SHALL use `context.get("artifacts", {}).get("skeleton_sources")` with fallback to current behavior (recompute or auto-generate). Registry unavailability (missing `.startd8/state/elements/`) SHALL degrade gracefully — log warning and skip all registry operations.

### NFR-MPA-004: Idempotency

Running plan ingestion twice with the same plan document SHALL produce byte-identical `skeleton_sources` and `element_tiers` artifacts (deterministic rendering + deterministic classification). Registry state may differ (second run has entries from first), but the seed output is deterministic for a given ForwardManifest.

### NFR-MPA-005: No Regression in Existing Paths

Tasks routed to Micro Prime (TRIVIAL/SIMPLE/MODERATE tiers) SHALL continue to work identically. Pre-assembly enhances the COMPLEX path and cross-run reuse without modifying the existing Micro Prime engine flow.

### NFR-MPA-006: Registry I/O Performance

Registry operations at ingestion SHALL complete within 500ms for registries containing up to 1,000 elements. If I/O exceeds this threshold, the pipeline SHALL log a warning and skip registry operations (fall through to current behavior).

## 11. Implementation Order

1. **FR-MPA-009** (registry population + manifest index) — establishes the element inventory; unblocks cross-run lookup
2. **FR-MPA-011** (context checksums) — required for staleness detection in cross-run lookup
3. **FR-MPA-010** (cross-run lookup + registry pre-fill) — enables W-4 elimination
4. **FR-MPA-001** (persist skeleton sources) — enables W-1 elimination
5. **FR-MPA-002** (element classification) — adds tier map to seed
6. **FR-MPA-003** (template pre-fill) — fills trivial bodies at ingestion; enables W-2 elimination
7. **FR-MPA-004** (SCAFFOLD consumption) — eliminates W-1
8. **FR-MPA-007** (Prime adapter forwarding) — eliminates recomputation in prime path
9. **FR-MPA-005 + FR-MPA-006** (IMPLEMENT prompt narrowing) — eliminates W-2 and W-3
10. **FR-MPA-008** (metrics) — measure the impact

Steps 1–3 wire the Element Registry at ingestion (one `plan_ingestion_workflow.py` change). Steps 4–6 add skeleton source and classification to the seed (same file, same EMIT function). Steps 7–9 are downstream consumption (separate files). Step 10 is observability.

All steps are independently shippable. Steps 1–6 can be implemented in a single PR since they all modify the same EMIT function.

## 12. Open Questions

1. **Seed size for large plans:** A 50-file plan could produce ~200 KB of skeleton source text. Should we compress (`zlib`) or store as a separate artifact file alongside the seed instead of inline JSON?

2. **Module inventory at ingestion:** SCAFFOLD discovers `module_inventory` by scanning the project. At ingestion time, we don't have this. Should we (a) accept less-precise import classification at ingestion, or (b) add a lightweight project scan to ingestion EMIT?

3. **Edit-mode prompt validation:** The prompt narrowing in FR-MPA-005/006 changes what the LLM sees. This needs A/B testing against current prompts to verify it doesn't degrade generation quality. Should we add a feature flag (`--pre-assembly-prompts`) to enable/disable?

4. **Interaction with `--force-design`:** When `--force-design` invalidates the design cache, should `skeleton_sources` also be invalidated and re-rendered? The skeletons are derived from ForwardManifest (not design docs), so they should be stable across design iterations.

5. **Registry scope at ingestion:** Should the ingestion-time registry use the same `.startd8/state/elements/` as the generation-time registry (shared state), or a separate ingestion-scoped store? Shared state enables cross-run reuse (the goal), but requires careful locking if ingestion and generation could run concurrently. Recommendation: shared state with `threading.RLock()` (already implemented in `ElementRegistry`).

6. **Element body vs. full definition in registry:** REQ-MP-11xx §8 Q3 asks whether to store body-only or full definition. For ingestion-time pre-fill, body-only is sufficient (the splicer handles indentation). But cross-task reuse in different skeleton contexts may need the full definition. Recommendation: store full definition, extract body at splice time.
