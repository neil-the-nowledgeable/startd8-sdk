## 1. Critical Insight: The Requirements Misdiagnosed the Root Cause

### What the requirements say

> "ForwardManifest file_specs have `elements: []` for non-Python files, so DFA assemblers have nothing to assemble."

### What planning revealed

**The DFA assemblers already handle empty elements.** All three assemblers (C#, Java, Node.js) have explicit fallback code:

```python
# csharp_file_assembler.py:328
if not type_elements:
    class_body = self._render_members(member_elements, class_name)
    sections.append(f"public class {class_name}\n{{\n{class_body}\n}}")
```

With `elements: []`, the C# assembler produces:
```csharp
// [STARTD8-SKELETON]
namespace CartService.CartStore;

public class CartStore
{
}
```

That's a correct namespace + class shell — exactly what REQ-DFA-102 (Minimal Skeleton) requires. **The assembler already does this.** It's called by prime_adapter (line 1861). The issue is `render_file()` returns this content, but the skeleton is then not used by the drafter because `skeleton_sources` isn't set in the context.

**The real root cause is not "elements are empty." It's "skeletons are produced but discarded."**

### Requirement correction

**REQ-DFA-108 (element deriver) is NOT the critical-path blocker.** The critical path is:

1. DFA assemblers produce skeletons (already happens for C#/Java/Node.js)
2. Skeletons are not injected into `context["skeleton_sources"]` (5-line fix)
3. Drafter doesn't see `skeleton_sources` → uses file-whole mode instead of skeleton_fill

**REQ-DFA-107 (skeleton_fill wiring) should be the Phase 0 requirement, not REQ-DFA-108.**

## 2. Quick Win Discovery: Three Zero-Effort Fixes

Planning revealed three fixes that require ≤10 lines each and deliver immediate value:

### Quick Win A: Inject Existing Skeletons into Context (~5 lines)

**File:** `prime_adapter.py`, after line 1863

The DFA assemblers are already called and produce valid skeletons. The skeletons are stored in the `skeletons` dict. But they're never injected into `context["skeleton_sources"]`.

```python
# After: skeletons[file_path] = cs_source
context.setdefault("skeleton_sources", {})[file_path] = cs_source
```

**Impact:** All C#/Java/Node.js files immediately use `skeleton_fill` mode. ~40% token reduction.

### Quick Win B: Don't Overwrite Deterministic Build Files (~10 lines)

**File:** `prime_contractor.py`

The PrimeContractor already calls `generate_dependency_file()` at line 3999 and writes the content to disk (e.g., `.csproj`, `go.mod`). Then the LLM **overwrites** it with its own version. The deterministic version is correct (includes `<Nullable>enable</Nullable>`). The LLM version has defects (missing Nullable in run-106).

Fix: When a feature's sole target file is a build file that was already generated deterministically, skip LLM generation for that feature.

```python
# In process_feature(), before LLM generation:
if all(self._is_deterministic_build_file(tf) for tf in feature.target_files):
    # Already generated by _generate_dependency_file_if_needed()
    return FeatureResult(success=True, cost=0.0, model="deterministic")
```

**Impact:** .csproj, go.mod, package.json, build.gradle stop going to LLM. $0.10-0.15 saved per build file. Run-106 had 2 build files ($0.22 wasted).

### Quick Win C: Interface Detection from Filename (~5 lines)

**File:** `csharp_file_assembler.py`, in `render_file()` around line 321

The C# assembler derives `class_name` from filename but always renders `public class`. For `ICartStore.cs`, it should render `public interface ICartStore`.

```python
class_name = _derive_class_name(file_path)
# Quick Win C: detect interface from I-prefix convention
is_interface = (class_name.startswith("I") and len(class_name) > 1
                and class_name[1].isupper())
type_keyword = "interface" if is_interface else "class"
```

**Impact:** ICartStore.cs gets correct `interface` keyword. Prevents the run-106 wrong-content defect. Zero NLP, zero contract parsing needed.

## 4. Robustness Improvements

### 4.1 Progressive Enrichment Tiers

The original requirements were binary: elements are populated or not. Planning revealed a **spectrum of enrichment** that should be explicitly defined:

| Tier | Input | Elements Produced | Skeleton Quality | Cost Savings |
|------|-------|-------------------|-----------------|-------------|
| **T0: Filename only** | `CartStore.cs` | `class CartStore` (empty body) | Namespace + class shell | ~20% (skeleton_fill overhead reduction) |
| **T1: + Description** | "Implements ICartStore" | `class CartStore : ICartStore` | + base class/interface | ~30% |
| **T2: + Contracts** | FM method signatures | + method stubs | + method shapes | ~45% |
| **T3: + Framework** | `framework_imports` detected | + DI constructor + imports | Full skeleton | ~60% |

**Requirement update:** REQ-DFA-108 should define these tiers explicitly. T0 is already working (assemblers handle empty elements). Each tier adds incremental value.

### 4.2 Degraded-Mode Behavior

The requirements assume rich feature descriptions. Real-world descriptions are often sparse ("Implement cart store"). The requirements should specify fallback behavior:

- **No description:** Use filename → T0 enrichment
- **No contracts:** Use filename + description keywords → T1 enrichment
- **No framework detection:** Use language defaults (e.g., C# always gets `System` imports) → T0-T1
- **Parser unavailable:** Use regex fallback (all 4 parsers have regex mode) → best-effort T1

### 4.3 Idempotency Requirement

**Missing from requirements:** The element deriver must be idempotent with the AST-based extractor:

- If `file_specs[path].elements` is already populated (from AST extraction of existing source in edit-mode), the deriver MUST NOT override it
- The deriver only runs when `elements: []` AND no existing source is available
- This prevents the deriver from producing less-accurate elements when better data (parsed AST) is available

### 4.4 Build File Routing Must Not Overwrite User Modifications

**Missing from requirements:** Quick Win B (don't overwrite deterministic build files) must handle the case where the user has manually edited the build file:

- If `.csproj` exists AND has been modified since the last run, don't overwrite with deterministic version
- Use file hash comparison: deterministic output hash vs. disk hash
- If hashes differ, the user modified it → preserve user version

## 5. Flexibility Improvements

### 5.1 Language-Specific Activation Ordering

The requirements treat all four languages equally. Planning revealed they have different readiness levels:

| Language | Readiness | Activation Effort | Reason |
|----------|-----------|-------------------|--------|
| **C#** | 95% | ~5 lines | DFA assembler exists, handles empty elements, produces valid skeletons with namespace + class |
| **Java** | 95% | ~5 lines | Same as C# — DFA handles empty elements |
| **Node.js** | 90% | ~5 lines | DFA handles empty elements; no splicer (skeleton_fill works without splicer) |
| **Go** | 60% | ~50 lines | Needs DFA assembler class; parser + splicer exist |

**Requirement update:** REQ-DFA-107 should specify activation in order: C# → Java → Node.js (Phase 0, same commit) → Go (Phase 1, new file).

### 5.2 Dockerfile Templates Should Be Language-Profile Methods

The plan proposed a standalone `dockerfile_generator.py`. But each language profile already has `docker_base_image` and `docker_runtime_image` properties, and `build_project_context_section()` already contains the Dockerfile guidance as literal text.

**Better approach:** Add `generate_dockerfile()` as an optional method on `LanguageProfile` protocol. Extract the literal Dockerfile text from `build_project_context_section()` into the generator. This keeps language-specific Docker knowledge co-located with the language profile.

### 5.3 Template Expansion Should Be Data-Driven

The requirements list specific templates to add (health check, config class, etc.). Planning revealed that the 29 existing templates are code-defined in `templates.py` as `CodeTemplate` objects.

**Better approach:** Templates should be loadable from YAML/JSON files alongside the language profile, not hardcoded in `templates.py`. This makes adding new templates a data change, not a code change.

**Not a Phase 0-2 requirement** — this is a future architectural improvement. But the requirements should note it as a design principle.
