##### Focus-area answers (sponsor asks)

**Focus 2 — Value & pipeline integration (the core R3 question): is Inc 9 deferral right, or must a minimal `provided_files` hook ship in v1?**
- **Summary answer:** The deferral as written *under-claims what already exists* — a minimal pipeline hook is **~30 lines, not Inc 9 surgery**, and without it the renderer delivers **zero prevention in a real run** (only a standalone artifact an operator must manually wire). Pull a thin hook into v1.
- **Rationale:** The "provided_files seam" the plan treats as deferred Phase C already exists in narrow form: `PrimeContractorWorkflow._try_deterministic_file_shortcut` (Phase 0.6, REQ-DFA-101a, `prime_contractor.py:3592-3639`) **already** marks a feature `GENERATED` at `$0.00`, skipping the LLM, when *all* its target files are deterministic build files that exist on disk — driven by the `_DETERMINISTIC_BUILD_NAMES`/`_DETERMINISTIC_BUILD_EXTENSIONS` frozensets (`prime_contractor.py:3584-3590`). The owned schema-types file (`lib/value-model.ts`) is the **identical pattern**: pre-write it, then a feature targeting it is skipped and never invented. The deferral conflates "exclude from LLM feature set" (hard — feature-list filtering) with "skip a feature whose owned file is already on disk" (already shipped). The latter is the actual RUN-011 fix and is in reach now.
- **Assumptions / conditions:** The owned file is pre-written to `output_dir` or `project_root` before Phase 0.6 runs (the shortcut already checks both, `prime_contractor.py:3623`); the renderer's predicate identifies owned files by name/regex rather than only hard-coded build names.
- **Suggested improvements:** R3-S1 (extend the shortcut to owned codegen outputs), R3-S2 (run the renderer in the pre-generation hook beside `_build_project_knowledge`), R3-F1 (split FR-8 into 8A-CLI / 8B-minimal-hook / 8C-full-ownership so v1 ships 8B).

**Focus: the owned/seeded split (FR-7) — earns its complexity, or ceremony?**
- **Summary answer:** **Owned** earns it (it is the RUN-011 fix); **seeded** is ceremony for v1 — it imports the unresolved OQ-4 clobber problem for zero RUN-011/012/013 value. Ship owned-only.
- **Rationale:** Every named invention class (RUN-011 schema fields, RUN-012 barrels/CSS, RUN-013 dirs) is an **owned** artifact. No postmortem invention is a *seeded route-shell body*. The seeded path (FR-7, Plan §8 Inc 7) only raises OQ-4 ("regenerable without clobbering the LLM body?") and OQ-5 (drift into consuming bodies) — both unresolved. R1-F8/R1-S? already flagged the minimal owned-only carve-out; R3 strengthens it to a sequencing mandate: seeded must not block v1.
- **Assumptions / conditions:** none.
- **Suggested improvements:** R3-S3 (move seeded shells out of Inc 7 into a deferred Inc 7b), R3-F2 (mark FR-7 seeded as v1-deferred, not just "minimal").

**Focus: CLI surface — `startd8 generate frontend` right, or a plan-ingestion hook?**
- **Summary answer:** Both, but **the hook is the value path and the CLI is the dev-loop path** — the doc presents the CLI as primary (FR-8A, Inc 8) and the hook as deferred, which inverts the value ordering.
- **Rationale:** `startd8 generate frontend` is a *new top-level command group* — `cli.py` is flat `@app.command()` decorators plus a handful of `add_typer` groups (`cli.py:740-760`); there is no `generate` group, so this is net-new surface. An operator running the *real* pipeline never types it; the prevention only fires if the renderer runs inside the run (the Phase 0.6 hook above). The CLI is valuable for CI/`--check` and local iteration, but should not be sold as the mechanism that "kills RUN-011 standalone" (Req §0, FR-1) — standalone it kills nothing in a live run.
- **Assumptions / conditions:** none.
- **Suggested improvements:** R3-S2, R3-S5 (`--check` is the CLI's real headline, per R1-S6/R1-F7 — R3 endorses and ties it to CI).

**Focus 5 — Sequencing: renderer-first (Inc 1–5) vs convention-detection-first (Inc 6)?**
- **Summary answer:** **Mis-sequenced.** FR-5 convention detection (Inc 6) supplies the `@/` alias the renderer needs to emit *correct import specifiers*, yet it lands *after* the renderer (Inc 3) and after the headline proof (Inc 5). Inc 5 "passes" precisely because strtd8's `value-model.ts` has no cross-module imports to get wrong — a hollow proof.
- **Rationale:** The codebase already resolved this dependency the other way: `project_knowledge.canonical_specifier` (`producer.py:44-55`) hardcodes `@/` because every consumer needs the alias to render an import. The renderer will face the same need the moment it emits anything beyond a single self-contained Zod file (enum imports, cross-schema refs, the `z` import path). Detecting the alias is a `tsconfig.json` `paths` read — cheap — and should precede or parallel Inc 3, not trail Inc 5.
- **Assumptions / conditions:** the renderer emits import specifiers (true once enums or shared types are emitted; R2's enum work makes this near-certain).
- **Suggested improvements:** R3-S4 (move alias detection ahead of Inc 3), R3-S7 (Inc 5 must include at least one cross-module-import case or be labeled a trivial-path proof).
