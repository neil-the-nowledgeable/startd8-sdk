## 1. Problem Statement

### 1.1 What Exists Today

The pipeline produces three observability artifact types:

| Artifact | Format | Purpose | Generated? |
|----------|--------|---------|-----------|
| Dashboard Spec | YAML → Grafana JSON | Operational metrics visualization | YES |
| Alert Rules | Prometheus YAML | Alerting on SLO violations | YES |
| SLO Definitions | OpenSLO v1 YAML | Error budget tracking | YES |
| **Onboarding Portal** | **???** | **"What is this project and how do I operate it?"** | **NO** |

The pipeline also produces `onboarding-metadata.json` with 41+ fields of rich project context: service communication graph, instrumentation hints, derivation rules, capability manifest, security contracts, SLO targets, alert configurations. This data is consumed by the code generation pipeline (plan ingestion, spec builder, drafter prompts) but is NOT presented to humans in a navigable format.

### 1.2 The Gap

Three hand-crafted HTML files exist (`harbor-tour-prime-contractor.html`, `developer-portal.html`, `harbor-tour.html`) that demonstrate what an onboarding portal looks like. These are manually authored, static, and not generated by the pipeline. They become stale as the project evolves.

The pipeline has all the data to generate these automatically. The missing piece is a **portal generator** — a fourth artifact type that reads `onboarding-metadata.json` and the three observability artifacts, and produces a navigable HTML view.

### 1.3 Value Proposition

| Audience | What They Need | Where It Comes From |
|----------|---------------|-------------------|
| **New engineer** | "What services exist? How do they communicate? Where are the dashboards?" | Service communication graph + dashboard spec paths |
| **On-call operator** | "What SLOs apply? What alerts can fire? What runbooks exist?" | SLO definitions + alert rules + annotations |
| **Engineering manager** | "What's the quality of this codebase? What's the security posture?" | Kaizen metrics + Security Prime scores + postmortem summary |
| **Platform team** | "What dependencies does this service have? What databases? What protocols?" | Instrumentation hints + security contracts + framework detection |

All of this data is IN the pipeline today. The portal makes it accessible without requiring `jq` and `cat` on JSON files.

## 2. Architecture

### 2.1 Portal as Fourth Artifact Type

The portal generator follows the same pattern as the existing three generators:

```
onboarding-metadata.json
    │
    ├── generate_alert_rules()      → alerts/     (existing)
    ├── generate_dashboard_spec()   → dashboards/ (existing)
    ├── generate_slo_definitions()  → slos/       (existing)
    └── generate_portal()           → portal/     (NEW)
```

The portal generator reads the SAME inputs (onboarding metadata + business context + manifest) and produces a DIFFERENT output format (HTML or Grafana dashboard).

### 2.2 Two Output Formats

| Format | Use Case | Toolchain | Dependency |
|--------|----------|-----------|-----------|
| **Static HTML** | Standalone, works anywhere, no Grafana needed | Template rendering (Jinja2 or f-strings) | None (browser only) |
| **Grafana Portal Dashboard** | Integrated with existing Grafana stack, linked from service dashboards | DashboardSpec YAML → Grafana JSON | Grafana instance |

**Recommendation:** Start with static HTML (zero dependencies, works offline, can be committed to repo). Add Grafana portal as a v2 enhancement.

## 3. Requirements

### REQ-OBP-100: Portal Content Sections

The generated portal MUST include the following sections, derived from pipeline context:

| Section | Source | Content |
|---------|--------|---------|
| **Project Overview** | `.contextcore.yaml` `spec.project` | Name, description, criticality, owner, repository |
| **Service Inventory** | `onboarding-metadata.json` `instrumentation_hints` | Service ID, transport (gRPC/HTTP), language, detected databases |
| **Service Communication Graph** | `onboarding-metadata.json` `service_communication_graph` | Visual or tabular representation of service-to-service calls |
| **SLO Summary** | Generated SLO artifacts | Per-service: availability target, latency P99, time window |
| **Alert Inventory** | Generated alert artifacts | Per-service: alert names, severity, thresholds |
| **Dashboard Links** | Generated dashboard specs | Per-service: link to dashboard (Grafana URL or file path) |
| **Security Posture** | `kaizen-metrics.json` `security` section | Aggregate security score, injection blocked, credential blocked |
| **Quality Metrics** | `kaizen-metrics.json` | Success rate, cost per feature, assembly delta |
| **Run Provenance** | `run-provenance.json` | Run ID, timestamp, pipeline version, duration |

### REQ-OBP-101: Portal Generation from Pipeline Context

| ID | Requirement |
|----|-------------|
| OBP-101a | The portal generator SHALL accept `onboarding-metadata.json`, optional `kaizen-metrics.json`, optional `run-provenance.json`, and the generated observability artifact directory as inputs. |
| OBP-101b | The portal SHALL be generated deterministically ($0.00 — no LLM calls). All content is derived from structured pipeline data. |
| OBP-101c | Missing optional inputs SHALL degrade gracefully — the portal omits the corresponding section rather than failing. |
| OBP-101d | The portal SHALL include a "Generated at" timestamp and "Pipeline version" for staleness detection. |

### REQ-OBP-102: Static HTML Output

| ID | Requirement |
|----|-------------|
| OBP-102a | The portal SHALL be a single self-contained HTML file (inline CSS, no external dependencies). |
| OBP-102b | The HTML SHALL use a dark theme consistent with the existing `harbor-tour-*.html` exemplars. |
| OBP-102c | The HTML SHALL be responsive (readable on desktop and mobile). |
| OBP-102d | Each section SHALL be navigable via a sidebar or top-level anchor links. |
| OBP-102e | The service communication graph SHOULD be rendered as an SVG or CSS grid (no JavaScript framework required). |

### REQ-OBP-103: Integration with Artifact Generator

| ID | Requirement |
|----|-------------|
| OBP-103a | `generate_observability_artifacts()` SHALL call `generate_portal()` as a fourth artifact type after dashboards, alerts, and SLOs. |
| OBP-103b | The portal result SHALL be an `ArtifactResult` with `artifact_type="portal"`, enabling quality validation via the same `_repair_and_validate()` pipeline. |
| OBP-103c | The portal SHALL be written to `portal/{project_id}-portal.html` in the output directory. |
| OBP-103d | The portal generation can be skipped via `--skip-portal` flag (opt-out, not opt-in). |

### REQ-OBP-104: Portal Validation (Kaizen)

| ID | Requirement |
|----|-------------|
| OBP-104a | Portal validation SHALL check: HTML parseable, all required sections present, service count matches instrumentation_hints count, SLO/alert sections populated when artifacts exist. |
| OBP-104b | Portal quality score: `checks_passed / checks_total` (same v1 pass-rate formula as other artifact types). |
| OBP-104c | Portal score SHALL be included in the per-service composite and the `observability_artifacts` section of `kaizen-metrics.json`. |

## 4. Content Derivation Rules

### 4.1 Project Overview

```
Source: .contextcore.yaml → spec.project
Fields:
  - name → Project title
  - id → Project identifier
  - description → One-paragraph summary (or "No description provided")
  - spec.business.criticality → Criticality badge (critical/high/medium/low)
  - spec.business.owner → Team owner
```

### 4.2 Service Inventory

```
Source: onboarding-metadata.json → instrumentation_hints
Per service:
  - service_id → Service name
  - transport → Protocol badge (gRPC / HTTP)
  - language → Language badge (detected from file extensions)
  - convention_metrics count → "N metrics instrumented"
  - detected_databases (from security contract) → Database badges
```

### 4.3 SLO Summary

```
Source: slos/{service}-slo.yaml (generated artifacts)
Per service:
  - spec.target → "99.9% availability"
  - spec.timeWindow.duration → "30-day rolling"
  - spec.indicator type → "Latency P99" / "Availability"
```

### 4.4 Alert Inventory

```
Source: alerts/{service}-alerts.yaml (generated artifacts)
Per service:
  - rules[].alert → Alert name
  - rules[].labels.severity → Severity badge
  - rules[].for → "fires after 5m"
  - rules[].annotations.summary → Description
```

## 5. Phased Delivery

### Phase 1: Generator (~150 lines)
- `generate_portal()` function in `artifact_generator.py`
- Static HTML template with f-string rendering
- Sections: Project Overview, Service Inventory, SLO Summary, Alert Inventory, Run Provenance
- Dark theme matching existing harbor tour exemplars

### Phase 2: Communication Graph (~50 lines)
- Parse `service_communication_graph` from onboarding metadata
- Render as CSS grid or simple table (no SVG dependency)

- Read `kaizen-metrics.json` `security` section for security posture

### Phase 4: Validation (~30 lines)
- Add portal checks to `observability_artifact_checks.py`
- Wire into `_repair_and_validate()` for `artifact_type="portal"`

## 6. Relationship to Existing Harbor Tour Files

The hand-crafted `harbor-tour-*.html` files are capability-level documentation (what Prime Contractor does, how it works). The generated portal is PROJECT-level documentation (what THIS project is, what services it has, what SLOs apply).

| Dimension | Harbor Tour (existing) | Generated Portal (this requirement) |
|-----------|----------------------|-------------------------------------|
| **Scope** | SDK capability documentation | Per-project operational overview |
| **Authoring** | Manual (human-written) | Generated (deterministic from pipeline data) |
| **Staleness** | Gets stale as SDK evolves | Regenerated every run |
| **Audience** | Evaluators, decision-makers | Operators, on-call engineers, new team members |
| **Data source** | Human knowledge | `onboarding-metadata.json` + generated artifacts |
