# Crossplane / Temporal Suitability Evaluation — Complement to Deterministic Generation

**Date:** 2026-06-07
**Status:** Evaluation (no adoption decision yet)
**Scope question:** Would Crossplane and/or Temporal complement the SDK's deterministic-first
generation strategy — across (A) SDK-internal orchestration, (B) generated-app runtime targets,
and (C) deployment/provisioning of generated apps — under the governing lens of
**"more platform robustness with less custom software to support"**?

---

## 1. Executive Summary

| Candidate | Verdict | One-line rationale |
|---|---|---|
| **Temporal — SDK-internal orchestration (Axis A)** | **CONDITIONAL YES — pilot as optional extra** | Replaces the genuinely fragile slice (resume, retry, run-state machines, batch waves) of our custom durability code; but the replaceable surface is ~4–6k lines of control-flow plumbing, **not** the ~117k-line orchestration stack, and it adds an external-process dependency to a pip-installable CLI. |
| **Temporal — generated-app runtime target (Axis B)** | **DEFER (conditional)** | Deterministically generating workflow/activity scaffolds would shrink bucket-3 LLM integration — but only for app domains with long-running/async flows. Its correct relationship to this SDK is as a future **output kind** (generate Compositions/Claims from the `.prisma` contract) if/when deploy scope opens — which is currently past bucket 3 and out of scope. |

## 2. Background: What Custom Software Are We Actually Supporting?

Inventory of the SDK's self-maintained orchestration/durability machinery.
**Reproducibility (R3-S5):** figures below are generated by the committed
`scripts/survey_orchestration_loc.py` (total physical lines of all `.py` files per subsystem);
re-run and diff at the decision gate — the verdict turns on them (2026-06-07 run):

| Subsystem | Files / lines | Nature |
|---|---|---|
| Contractors (checkpoint, queue, prime_contractor, integration_engine, batch_postmortem, …) | 75 files / 68,064 | Mostly **domain logic** |
| Repair (orchestrator, staging, retry pipeline) | 74 files / 12,083 | Mixed domain + control flow |
| Workflows (base, registry, builtin) | 63 files / 30,169 | Mostly domain logic |
| Resilience + ratelimit | 4 files / 1,129 | **Pure durability plumbing** |
| Events (bus, types) | 5 files / 1,038 | Coordination plumbing |
| backend_codegen (deterministic output) | 15 files / 4,866 | Generation — not orchestration |
| **Total** | **236 files / 117,349 lines** | |

**The critical distinction this evaluation turns on:** a durable-execution engine replaces
*durability and control-flow plumbing*, not domain logic. Decomposing the inventory:

**Temporal-replaceable (plumbing we wrote because nothing else held our state):**
- `queue.py` resume-state persistence + atomic `.prime_contractor_state.json` writes (~half of 762 lines; the cycle-detection half stays — it's domain validation, and LLM dependency graphs are unreliable regardless of engine)
- `resilience/` retry presets + circuit breaker (~287 lines) → Temporal RetryPolicy per activity
- `ratelimit/` token buckets (~842 lines) → partially replaced by Temporal task-queue rate limiting (token-count limits would remain ours). **Composition rule required (R4-S1):** exactly one limiter authoritative per dimension — task-queue limits own request/concurrency throttling, the retained token bucket owns tokens/min — so the two never stack into double-throttle or mutual starvation
- Batch run-snapshot accumulation in `batch_postmortem.py` (the run-tracking spine, not the analytics) → workflow history + queries
- Repair-loop retry control flow + per-category circuit-breaker global dict (`_circuit_breaker_state` — a known fragility: process-global, reset only explicitly)
- Crash-recovery semantics around mid-run failure (snapshot/rollback choreography in `integration_engine.py` would remain, but *when to re-enter it* becomes engine-driven replay instead of hand-rolled resume)

**NOT Temporal-replaceable (domain logic that stays regardless):**
- `checkpoint.py`'s 6-stage validation pipeline (1,192 lines — it's *validation*, not durability)
- Merge strategies, repair steps, language profiles, manifest/contract validation
- All of `backend_codegen/` and the Kaizen/postmortem analytics
- Cycle detection/breaking (the "aggressive deadlock breaking" in queue.py is a domain decision)

**Honest sizing: ~4–6k lines of genuinely fragile plumbing are in play**, plus the harder-to-count
cost of every future durability feature we *won't* have to write (waves, heartbeats,
long-run resume, cross-process visibility). The remaining ~111k lines are unaffected either way.

Known fragility signals the inventory surfaced in exactly the replaceable slice:
- Repair circuit-breaker state is a process-global dict
- Event-history trimming has a documented (low-probability) race
- Resume correctness depends on three hand-rolled layers (schema version → seed checksum → per-task hash)
- `checkpoint-v4 wave_*` is a **verified phantom** (per batch-orchestration review) — wave scheduling was designed but never actually implemented; Temporal child workflows would be the *real* implementation rather than an extension of phantom machinery

## 3. Candidates at a Glance (verified 2026-06-07)

| | Temporal | Crossplane |
|---|---|---|
| **What it is** | Durable-execution engine: workflows survive process death via event-sourced replay; activities get retries, timeouts, heartbeats | Kubernetes-based control plane: declarative desired-state reconciliation of (cloud) resources via Compositions + composition functions |
| **License** | MIT (server + all SDKs) | Apache-2.0 |
| **Governance** | Temporal Technologies (open core; cloud is the paid product, self-host fully free) | CNCF **Graduated** (Oct 2025) |
| **Python story** | Mature first-class SDK (`temporalio` on PyPI), actively maintained, sandboxed deterministic workflows | `function-python` for composition functions; consumers author YAML/Python functions |
| **Dev footprint** | `temporal server start-dev` — **single binary, zero runtime deps, SQLite persistence**. ⚠ In-memory by **default**; durable resume requires on-disk `--db-filename` (see §8.1 persistence mandate) | Requires a Kubernetes cluster (kind/minikube locally) |
| **Prod footprint (self-host)** | Server node ~4 vCPU/8 GB + Postgres/Cassandra (+optional ES); ~$400–900/mo cloud-equivalent | A managed/maintained K8s cluster + providers |
| **Determinism model** | Workflow code must be deterministic (no wall-clock/random/IO in workflow scope); side effects live in activities | Reconciliation loops converge on declared state; functions are deterministic transforms |
| **Notable recent** | — | v2: namespaced-first, composes *arbitrary* K8s resources; alpha **Operations/CronOperation/WatchOperation** for day-2 tasks |

A philosophical note that matters for fit — stated precisely (R3-S6): Temporal's
workflow-determinism constraint and Crossplane's *composition functions as pure transforms* are
the same bet this SDK makes with the $0 cascade — deterministic skeleton, effects pushed to the
edges. But the alignment claim is scoped to **generation**: Crossplane's *runtime* reconciliation
is eventually-consistent, **not** deterministic. Generating Compositions deterministically is
aligned; expecting deterministic deploys from the control loop would be a misreading.

## 4. Axis A — SDK-Internal Orchestration (Temporal)

### The fit

A `PrimeContractorWorkflow` run is *exactly* the workload durable execution was built for:
long-running (minutes–hours), many expensive fallible side effects (LLM calls), staged
checkpoints, retries with backoff, mid-run crash recovery, and multi-run batch progression.

Mapping (no rewrite of domain logic — the engine swaps in *under* the existing protocol seams):

| Today (custom) | Under Temporal |
|---|---|
| Run loop in `prime_contractor.py` driving feature queue | Workflow function (deterministic orchestration only) |
| LLM generation call, checkpoint subprocess runs, merge, repair step | **Activities** — with per-activity `start_to_close`/heartbeat tuned for **multi-minute LLM calls** (R3-S2: SDK-default timeouts would kill and retry mid-call); cooperative cancellation threaded into the `httpx` request so cancel/timeout actually stops the billed call (R5-S2); cost-metering idempotency keys so retries never double-count spend in existing dashboards (R2-S3); `maximum_attempts` poison caps composing with the retained domain circuit breaker — each failure counted once (R4-S5) |
| *(no file-state equivalent)* Workflow identity | Deterministic workflow IDs derived from batch/run ID + an explicit `WorkflowIDReusePolicy`, so re-launching the same logical batch dedupes/rejects instead of forking a duplicate durable (and double-billed) run (R6-S6) |
| `.prime_contractor_state.json` + 3-layer resume validation | Event-sourced history; resume is replay — delete the custom layers |
| `resilience/` presets, per-call retry loops | `RetryPolicy` declared per activity |
| Repair retry pipeline control flow + circuit breaker dict | Activity retry + workflow-scoped state (no process-global) |
| Batch ledger run snapshots; phantom `wave_*` | Parent workflow + child workflows per wave; queries for live progress |
| EventBus for cross-process visibility | Workflow queries/signals; telemetry exported through the **existing OTel pipeline** (Loki/Tempo/Grafana) via temporalio interceptors, with `workflow_id` ↔ run-ID correlation — Temporal UI is supplementary, **not** a second system of record (R1-S6) |

The SDK already has the seams for this: `CodeGenerator`, `MergeStrategy`, `Instrumentor`,
`IntegrationListener` protocols mean a `TemporalRunDriver` can be an alternative *driver*
around unchanged domain components. **Caveat (R1-S1): no *driver-level* seam exists today** —
the run loop is embedded in `prime_contractor.py`. Extracting a `RunDriver` protocol and
putting the existing file-state path behind it (test suite green) is **Step 0** of any spike;
if that extraction proves hard, the "no rewrite of domain logic" premise is falsified at $0
cost, before any temporalio code is written.

**Second caveat (R6-S3):** Temporal's workflow sandbox re-imports workflow-scope modules and
restricts non-deterministic ones, while the SDK does registry/entry-point discovery at import
time (the `ProviderRegistry.discover()` / `LanguageRegistry.discover()` pattern). Domain
components must be imported in *activity* scope (or via documented sandbox passthrough) — a
second hidden cost to the "no rewrite" premise that Step 0 must surface.

**Tracked upside (R6-S7):** the architecture's human bookends (security gate, convergent
review, DATA MODEL/RETROSPECTIVE) map naturally to Temporal **signals** — durable
human-in-the-loop pause/resume that the file-state path implements ad hoc. Recorded as value
evidence for the decision gate, not spike scope.

### The costs (why this is CONDITIONAL, not YES)

1. **Distribution model tension.** The SDK is a pip-installable CLI for single-developer local
   runs. Temporal requires a server process. The dev server (single binary, SQLite) makes this
   tolerable — `startd8` could auto-spawn it like it manages other subprocesses — but it is a new
   external binary to install, version, and support. *Every* support ticket gains a "is the
   Temporal server running/healthy?" branch. **Accounting rule (R7-S4):** server *supervision*
   (health, restart, orphan reap) is materially more than the SDK's existing fire-and-wait
   subprocess runs; the lifecycle spec must state whether it reuses existing subprocess
   machinery or is net-new code — net-new supervision LOC counts *against* the governing lens
   in the decision-gate ledger (R6-S1).
2. **Workflow determinism constraints.** Orchestration code moves into a sandbox forbidding
   wall-clock, randomness, and direct IO. Our run loop currently interleaves these freely;
   extracting a deterministic spine is real (if healthy) refactoring work. **Concrete repo
   collision (R6-S2):** the `get_logger`-for-Loki Must-Do cannot apply in workflow scope —
   logs there replay and duplicate. Policy: Temporal's replay-aware logger in workflow scope,
   `get_logger` (Loki bridge) inside activities only; the logger-acquisition policy test
   allowlist must encode this split.
3. **Versioning discipline.** In-flight workflow histories must replay against current code;
   Temporal's patching/versioning API is a new skill the project must carry.
4. **Replaceable surface is modest.** ~4–6k lines. The win is less "delete code today" and more
   "stop *growing* this category" — batch orchestration (waves, multi-run resume, parallel
   scheduling) is the next major consumer of durability plumbing, and it is currently specced
   against machinery that is partly phantom.
5. **The current machinery works.** Checkpoint/resume shipped and is battle-tested across runs.
   This is a replacement of working code, justified only by the *forward* cost curve.
   "Battle-tested" is currently **unquantified (R4-S3)**: a pre-spike baseline (today's resume
   success rate + enumerated known failure cases) must be captured so the scorecard has a real
   comparator instead of an assumed 100%.

### Verdict A

**Pilot, behind an extra.** `pip install startd8[temporal]`; file-state path remains the default
and the fallback. The pilot target should be **Plan Batch Orchestration** (Increment ≥1), not a
retrofit of the single-run path — that's where new durability code would otherwise be written,
so Temporal competes against *unwritten* code (cheapest possible adoption point) rather than
against shipped, working code.

## 5. Axis B — Generated-App Runtime Target (Temporal)

The idea: make Temporal workflow/activity scaffolds an *output kind* of the $0 cascade —
deterministically generate the durable spine for an app's long-running flows from the contract,
shrinking the bucket-3 LLM integration surface (the one in-scope LLM aspect).

**Assessment:** structurally attractive, currently premature.

- The cascade's target apps today are server-rendered CRUD (FastAPI+Pydantic+HTMX+SQLModel,
  12 owned kinds). The real strtd8 15-model contract has essentially **no long-running/async
  flows** that need durable execution. Generating Temporal scaffolds would add an
  infrastructure dependency to every generated app for a capability the app class doesn't use.
- The leverage condition: when a commissioned app's data model implies durable processes
  (multi-step fulfillment, human-in-the-loop approvals, scheduled pipelines, saga-style
  cross-service writes), an LLM today would hand-write that glue in bucket 3 — the
  most failure-prone code class there is. A deterministic `workflows.py`/`activities.py`
  projection from contract annotations would be a high-value new owned kind.
- Prereq: a contract-level way to *express* "this is a durable process" (e.g., schema
  annotations or a manifest extension), which doesn't exist yet and shouldn't be invented
  ahead of a real app that needs it.

**Verdict B: DEFER.** Re-open when a target app's requirements include durable processes.
Record as a candidate owned kind so it's evaluated per-app, not forgotten.

## 6. Axis C — Deployment/Provisioning of Generated Apps (Crossplane)

### Why Crossplane doesn't fit Axes A/B

Crossplane reconciles *desired resource state*; it does not execute imperative multi-step
pipelines. Driving codegen runs with it (Axis A) would mean modeling LLM passes as reconciled
resources — a category error (the alpha Operations feature edges toward tasks, but it is alpha,
K8s-bound, and far weaker than Temporal for this). As a generated runtime component (Axis B)
it only makes sense for apps that are themselves K8s platforms — not our app class.

### Where it does fit: a future output target

The genuine alignment is **contract → infrastructure projection**:

- The cascade today stops at `requirements.txt` — **zero deploy artifacts are generated**
  (no Dockerfile, no compose, no K8s manifests; confirmed in survey). Deployment is past
  bucket 3 and deliberately out of scope.
- The architecture memory says generated apps are "modular-monolith-**microservice-ready**."
  If/when that promise is exercised, the deploy story becomes: each module needs a database,
  a runtime, networking. Crossplane v2 (namespaced, arbitrary-resource composition, Python
  functions) is precisely a declarative target for *deterministic projection* of
  `schema.prisma` + module boundaries → `Composition`/`Claim` YAML. That projection is a $0
  deterministic output kind, philosophically identical to what `backend_codegen` already does —
  the determinism lives in the generation, **not** in the runtime reconciliation (R3-S6; §3).
  **Secret-reference convention (R7-S2):** any future projection emits infra secrets (DB creds,
  API keys) as *references* (ExternalSecrets/SealedSecrets/connection-secret refs), never
  inlined values — the Axis-C analogue of the workflow-history secrets policy; validated by a
  zero-inline-secrets scan of generated YAML when the axis opens.
- Until then, a single FastAPI+SQLite/Postgres monolith needs at most a generated Dockerfile +
  compose file — and adopting a K8s control plane to deploy that would *increase* the platform's
  supported-software surface, directly violating the governing lens.

**Verdict C: NOT NOW; TRACK.** When deploy scope opens, sequence is: (1) generated
Dockerfile/compose as new owned kinds (near-zero cost, no new platform deps), (2) Crossplane
Compositions as the K8s-tier projection only when a commissioning company actually deploys to
K8s. Crossplane enters as *generated output*, never as an SDK runtime dependency.

**Re-open triggers (R2-S7)** — TRACK leaves rot mode only with observable gates; Axis C
re-opens when any of these fires:
1. a commissioning company mandates Kubernetes as the deployment target;
2. a generated app splits into >1 independently deployable module;
3. a generated app requires provisioned/managed infrastructure (managed Postgres, object
   storage, queues) beyond what compose-on-a-VM covers.

Tracking hook (zero implementation): register **Dockerfile + compose** as named candidate owned
kinds in the backend_codegen candidate list with "trigger = deploy scope."

## 7. Suitability Matrix

Scored against the governing lens. ✔ = strong, ◐ = partial, ✘ = poor, — = not applicable.
The **harden-in-place baseline column (R3-S1)** scores the cheapest honest alternative: fix the
three named §2 fragilities (~200 LOC: instance-scoped circuit breaker, trim-race fix, resume
hardening) with no new engine — adoption must state why the engine beats it.

| Criterion | Harden in place (baseline) | Temporal (Axis A) | Temporal (Axis B) | Crossplane (Axis C) |
|---|---|---|---|---|
| Replaces custom software we currently support | ◐ (fixes ~200 fragile LOC; retires nothing) | ✔ (resilience, resume, batch spine) | ✘ (adds a new kind) | ✘ today / ◐ future (replaces unwritten deploy glue) |
| Reduces *future* custom software | ✘ (batch durability still hand-written) | ✔✔ (batch orchestration unwritten) | ✔ (bucket-3 glue, conditionally) | ◐ (deploy tier, conditionally) |
| Robustness gain | ◐ (named fragilities only; resume stays 3-layer) | ✔ (replay > 3-layer hand-rolled resume; kills global-dict circuit breaker) | ◐ | ◐ |
| Fits pip-installable single-dev CLI distribution | ✔✔ (zero new deps) | ◐ (dev server single binary; still an external process) | ✘ for current app class | ✘ (requires K8s cluster) |
| Fits deterministic-first philosophy | — | ✔ (deterministic workflow core) | ✔ (deterministic projection) | ◐ (generation-side only — runtime reconciliation is eventually-consistent, §3/R3-S6) |
| License/governance | — | ✔ MIT | ✔ MIT | ✔ Apache-2.0, CNCF Graduated |
| Maturity of the path we'd use | ✔ (in-house, known code) | ✔ (Python SDK mature) | ✔ | ◐ (v2 recent; Operations alpha) |
| Adoption cost vs. value now | ✔ cheapest now / ✘ forward cost curve unchanged | ◐ (refactor + new binary vs ~4–6k lines + future) | ✘ | ✘ |

**Mid-weight tier dismissed (R5-S6)** — so "full engine" wins on merit, not false dichotomy.
Between harden-in-place and Temporal sit SQLite-backed durable job queues and scheduler
libraries (dramatiq-, huey-, apscheduler-class). They give task retry/persistence but: **no
event-sourced replay** (resume re-runs work rather than replaying state — the silent-poison
class returns), **no deterministic-workflow model** (mid-orchestration crash recovery stays
hand-rolled — the exact plumbing this evaluation is about not writing), and **no
child-workflow/wave primitives** (batch waves remain custom). They replace the easy third of
the plumbing while keeping the hard two-thirds.

## 8. Recommendation & Phasing

*Approval gate (R5-S7): nothing below is green-lit by this document — the Status line stands.
The spike starts only on maintainer sign-off at the decision gate in item 1; the gate's outcome
is recorded as an ADR carrying all verdicts and their re-open triggers (R7-S5).*

1. **Now — Temporal spike (Axis A), scoped to batch orchestration.**
   - **Charter bounds (R5-S5):** explicit timebox (calendar days) and LLM-spend ceiling, set
     when the gate opens; overrun forces a go/no-go review — a falsified premise ends the
     spike at known cost, never drifts into a project.
   - **Step 0 (R1-S1):** extract a `RunDriver` protocol and put the existing file-state run
     loop behind it, current test suite green — *before* any temporalio code. If extraction
     is hard, the spike's core premise is falsified at $0. Step 0 also surfaces the sandbox
     question (R6-S3): import-time-discovery modules go in activity scope or documented
     passthrough.
   - Prototype: batch parent workflow + per-wave child workflows + LLM-call/checkpoint/repair
     activities, driving the *existing* domain components through the existing protocol seams.
     Run against the strtd8 pilot batch; compare against the BatchLedger-extension path.
   - **Persistence mandate (R1-S9):** dev server runs with on-disk `--db-filename` under
     `.startd8/temporal/` — in-memory is the upstream default and would make replay-resume
     illusory for exactly the host/process-death crashes the adoption is justified by. Resume
     after host death = re-spawn server against the same DB file, then replay.
   - **Engagement policy (R2-S2):** Temporal is opt-in. With the extra installed but the
     server unreachable at run start: warn and fall back to file-state, unless
     `--require-temporal` (then exit non-zero).
   - **Runtime lifecycle deliverable (R1-S7 + R5-S1 + R2-S5 + R6-S6 + R7-S4):** one spec
     covering the **server** (DB location, port-conflict handling, spawn/shutdown ownership,
     orphan cleanup, `temporal` CLI binary pinning + install story — not pip-installable) and
     the **worker** (in-process with the CLI vs daemon; how host death + respawn re-attaches a
     worker so in-flight *activities* — not just the schedule — resume; durable scheduling
     without a running worker is a run that never progresses); task-queue isolation per
     project root (two repos sharing one dev DB must not cross-deliver tasks); deterministic
     workflow IDs + `WorkflowIDReusePolicy`; and an explicit reuse-vs-net-new statement for
     subprocess supervision code (net-new LOC goes in the gate ledger).
   - **Activity-semantics deliverables:** typed, **versioned** JSON-serializable Pydantic
     contracts per boundary with backward-compat rules for history payloads spanning an SDK
     upgrade (R1-S3 + R4-S6 — Keiyaku/REQ-MP-1010: activity args/results are A2A payloads
     persisted into history); per-activity `start_to_close` + heartbeat tuned for multi-minute
     LLM calls (R3-S2); cooperative cancellation threaded into the `httpx` request, with
     SIGINT/Ctrl+C semantics and bounded staging cleanup on both engines (R5-S2 + R2-S4);
     poison caps composing with the retained domain breaker (R4-S5); cost-metering idempotency
     keys (R2-S3).
   - **Observability deliverable (R1-S6 + R6-S5 + R6-S2):** temporalio interceptors export
     through the existing Loki/Tempo/Grafana stack with `workflow_id` ↔ run-ID correlation —
     via span attributes/exemplars, **never** Prometheus metric labels (Mimir cardinality);
     replay-aware logger in workflow scope, `get_logger` inside activities only; Temporal UI
     stays supplementary, not the system of record.
   - **Determinism CI guard (R3-S3):** lint/AST or replay-test job failing any wall-clock /
     `random` / direct-IO / `get_logger` / import-time-discovery usage in workflow scope — the
     enforcement point for R6-S2/R6-S3. A planted `datetime.now()` must fail the build, not a
     customer's resume.
   - **Decision gate (R1-S2):** the spike is pinned to the next Plan Batch Orchestration
     increment that would otherwise extend `BatchLedger` durability; that extension work is
     **held while the gate is open**, with a decision-by date set when the increment is
     scheduled — otherwise the "competes against unwritten code" premise decays with every
     increment shipped on the custom path. **Gate inputs (R6-S1 + R3-S1):** the *sized*
     durability LOC + feature list the increment would add on the custom path (the actual
     unwritten code Temporal is bid against), and the scored harden-in-place baseline (§7) —
     adoption must state why the engine beats both. (Cross-reference to be added to
     `docs/design/CRP_PLAN_BATCH_ORCHESTRATION_*` when that increment is planned.)
   - Ship as `pip install startd8[temporal]`; file-state path remains default and fallback.
   - **Exit criterion — numeric scorecard (R1-S5), template committed before the spike
     starts; adoption cites the filled scorecard, not anecdote:**
     - **Row 0 — pre-spike baseline (R4-S3):** today's resume success rate + enumerated known
       failure cases, captured before any temporalio code; every row below compares against
       it, not against an assumed 100%.
     - **Dual-driver parity (R2-S1):** the same pilot batch fixture end-to-end on both
       drivers — kaizen scores, disk compliance, feature success counts within stated
       tolerance; a parity miss blocks adoption regardless of resume results.
     - ≥10 injected kills at randomized pipeline stages, 100% correct resume — including
       server respawn after host death (R1-S9), **worker kill with activity re-attach
       (R5-S1)**, the mutated-staging-artifact case (must fail **loudly** at dereference,
       R1-S4), and a corrupted-dev-DB case behaving per the documented blast-radius policy
       (R4-S7)
     - plumbing-LOC-retired measured via the committed §2 survey method (R3-S5)
     - dev-server setup failure rate below a stated threshold across fresh machines (answers OQ-3)
     - per-activity orchestration overhead within a stated budget
     - zero secret patterns in an exported-history scan (R1-S8)
     - history bounded via continue-as-new on a batch ≥2× the pilot wave count (R2-S6/OQ-6)
     - **Harness (R4-S4 + R5-S4):** kill/timeout cases run on Temporal's time-skipping test
       framework where possible (no wall-clock sleeps in CI); the real-process-death and
       worker-re-attach rows run against a CI-provisioned real server (service container or
       cached binary), wired into the OQ-5 matrix job.
2. **Not now — Axis B.** Add "durable-process owned kind (Temporal scaffolds)" to the

3. **Not now — Axis C.** Trigger = the §6 re-open trigger table (R2-S7). First step then is
   Dockerfile/compose owned kinds — with golden-file + `--check` drift tests mirroring

4. **Do not** adopt Crossplane Operations as a Temporal alternative for Axis A (alpha,

## 9. Risks & Open Questions

- **OQ-1:** Can the prime-contractor run loop's deterministic spine be extracted without
  destabilizing the shipped single-run path? (Spike answers this for the batch path first.)
- **OQ-2:** Temporal payload limits (~2 MB/event) vs. our artifact sizes — large generated
  files must pass by reference (paths/staging dirs), which the staging system already supports.
  **Constraint (R1-S4):** every dereference of a by-reference path from workflow history must be
  integrity-checked (existence + checksum, or content-addressed staging) — histories don't
  version filesystem state, so replay could otherwise "succeed" against stale/mutated artifacts:
  a silent-poison failure mode *worse* than today's checksummed 3-layer resume.
- **OQ-3:** Does auto-spawning the dev server meet the "less software to support" bar, or does
  it just relocate the support burden? Measured by the §8.1 scorecard (setup failure rate,
  version skew) plus the R1-S7 lifecycle spec and the R7-S4 reuse-vs-net-new accounting.
- **OQ-4:** Workflow versioning policy for long-lived batches spanning SDK upgrades.
  **Extended (R4-S6):** covers *payload* schema evolution too — versioned activity contracts
  with backward-compat deserialization rules (round-trip test: a payload serialized by model
  vN replays under vN+1 — added optional field, aliased rename — successfully or fails loudly;
  a field rename mid-batch silently breaking resume is the exact crash class adoption is
  justified by).
- **OQ-5 (R1-S10):** `temporalio` ships Rust-core wheels with its own Python/platform floor;
  the SDK declares Python 3.9+ (venv runs 3.14). CI must install `startd8[temporal]` across the
  declared matrix; any narrowing is documented as an explicit constraint of the *extra*, never
  silently leaked into core support promises.
- **OQ-6 (R2-S6):** parent workflow **continue-as-new per wave** to bound history growth on
  long batches — OQ-2 covers per-event payload size, not unbounded multi-wave history;
  demonstrated in the scorecard on a ≥2× pilot wave count.
- **Reversibility — both ramps (R3-S4 + R5-S3):** *exit* — if the pilot is abandoned,
  in-flight Temporal histories are drained (run to completion, or documented file-state
  handoff) within a stated max in-flight window before the extra can be safely removed;
  *entry* — enabling the extra never migrates an in-flight `.prime_contractor_state.json` run:
  it finishes on its original engine or aborts cleanly with a warning. A single run is never
  half file-state, half Temporal.
- **Corruption blast radius (R4-S7):** the on-disk dev DB concentrates *all* in-flight run
  state in one SQLite file — a single corruption point the per-feature file-state checkpoints
  don't have. The recovery/rebuild story is documented (or the blast radius explicitly
  accepted); a corrupt-DB resume must fail clean, never partially replay.
- **Supply chain (R6-S4):** `temporalio` ships a Rust core + transitive deps. Before the extra
  ships: CI-generated SBOM + advisory scan gating its release, native-wheel provenance
  documented. (OQ-5 is compatibility; this is the trust surface — "optional" still ships in
  users' environments.)
- **Security (R1-S8):** workflow history is a **new at-rest store outside `.startd8/`** for
  activity inputs/outputs — i.e., prompts, generated code, and potentially
  commissioning-company content. Policy: API keys are read from env *inside activities only*
  and never appear in workflow scope, activity args, or history; large/sensitive payloads pass
  by reference (per OQ-2); the dev-server DB gets a documented retention/cleanup policy.
  Verified by the scorecard's exported-history secret-pattern scan.
- **Risk:** open-core gravity — Temporal's paid pull is its cloud; self-host MIT is complete
  today, but monitor for feature bifurcation.
