# Kaizen Data Analysis Guide

This guide explains how to locate, interpret, and use the Kaizen telemetry data generated by Prime Contractor runs. Kaizen is a closed-loop continuous improvement system that captures raw LLM prompts, tracks success rates, and correlates prompt characteristics to outcomes.

## 1. Enabling Kaizen Capture

To enable Kaizen prompt capture and metric generation, you must explicitly pass the `--kaizen` flag when running the `run_prime_workflow.py` CLI:

```bash
python scripts/run_prime_workflow.py --kaizen ...
```

By default, Kaizen data is written to the execution's output directory (often inside `.cap-dev-pipe/pipeline-output/[project_name]/[run_id]/`). There will be sub-directories per phase, such as `plan-ingestion`, `verification`, etc.

## 2. Directory Structure & Finding the Data

Within the run's phase directory (e.g. `run-XYZ/plan-ingestion/`), Kaizen outputs are split into two levels:

1. **Aggregated Phase Telemetry**:
   - `kaizen-metrics.json`: High-level metrics for the phase run.
   - `kaizen-analysis.log`: Execution logs for the post-mortem analysis scripts.
   - `kaizen-suggestions.json`: Automatically generated prompt improvement suggestions.
   - `kaizen-correlation.md` / `kaizen-trends.md`: Markdown reports on prompt characteristics vs success metrics.

2. **Per-Feature Prompt Data** (`kaizen-prompts/`):
   Contains subdirectories for each feature generated (e.g. `standalone/PI-005/`), governed by the `KAIZEN_RUN_ID` and feature ID.

## 3. Reading the Per-Feature Prompts

Inside `kaizen-prompts/standalone/<FEATURE_ID>/`, you will find the exact artifacts sent to and received from the LLM.

### Metadata

- **`metadata.json`**: Documents the exact agent models used, the target files, boolean flags (like `has_existing_files`), and the full list of context keys that were active during prompt generation.

### The Prompt & Response Pairs

The Lead Contractor workflow runs three main phases per feature. You'll see corresponding files for each:

1. **Spec Phase** (`spec_user_prompt.md`, `spec_system_prompt.md`)
2. **Draft Phase** (`draft_user_prompt.md`, `draft_system_prompt.md`)
3. **Review Phase** (`review_user_prompt.md`, `review_system_prompt.md`)

Each phase also generates a captured LLM output:

- **`{phase}_response.md`**: The raw text output from the LLM. *Any configured PII redactions are applied before this file is written.*
- **`{phase}_response.meta.json`**: Describes if the LLM output exceeded the 2 MiB capture threshold and got truncated.

> [!WARNING]
> **Missing Response Files**: If you see user/system prompts but **no** response file (e.g., `draft_response.md` is missing), this indicates the LLM generation process suffered a hard fault—such as a network timeout, an empty response from Ollama, or an unrecoverable structural syntax crash before the response could be persisted.

## 4. Analyzing Aggregated Metrics

### `kaizen-metrics.json`

This is the core health check of the run. Look for:

- `success_rate`: The ratio of PASS features to total features.
- `top_root_causes`: If failures occurred, what layer threw the block? (e.g., `generation_error`, `repair_exhausted`, `validation_failure`).
- `pipeline_attribution`: Locates exactly where in the pipeline the fault happened (e.g. `ollama_generation`).

### `kaizen-correlation.md`

This markdown file maps scalar characteristics (like feature sizes, prompt word counts, number of dependencies) to the PASS/FAIL groups.

- Look at the `PASS mean` vs `FAIL mean` columns.
- If the `FAIL mean` for `target_file_count` is consistently much higher than the `PASS mean`, it indicates your Micro Prime decomposer is failing to split large chunks of work effectively.

### `kaizen-suggestions.json`

Consolidated lessons-learned generated by the analysis engine. If Kaizen notices repeated failures heavily correlated with certain prompt structures (e.g., missing imports or missing return types), it will template a suggestion here to be injected into the next run via `--kaizen-config`.

## 5. Typical Debugging Workflow

1. You notice a feature failed during a run.
2. Open `kaizen-metrics.json` to identify the `top_root_causes`.
3. If it’s a generation fault, navigate to `kaizen-prompts/standalone/<FEATURE_ID>/`.
4. Inspect `draft_response.md` to see exactly what the LLM produced.
5. If the LLM hallucinated, open `draft_user_prompt.md` to cross-reference what instructions it was given. Was the necessary context missing?
6. Create an experimental hint in your `kaizen-config.json` targeting that feature, rerun with `--kaizen-config`, and observe if the new hint corrects the LLM behavior.
