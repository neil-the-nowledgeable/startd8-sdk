# Kaizen Data Analysis Guide

This guide explains how to locate, interpret, and use the Kaizen telemetry data generated by Prime Contractor runs. Kaizen is a closed-loop continuous improvement system that captures raw LLM prompts, tracks success rates, and correlates prompt characteristics to outcomes.

## 1. Enabling Kaizen Capture

To enable Kaizen prompt capture and metric generation, you must explicitly pass the `--kaizen` flag when running the `run_prime_workflow.py` CLI:

```bash
python scripts/run_prime_workflow.py --kaizen ...
```

By default, Kaizen data is written to the execution's output directory (often inside `.cap-dev-pipe/pipeline-output/[project_name]/[run_id]/`). There will be sub-directories per phase, such as `plan-ingestion`, `verification`, etc.

## 2. Directory Structure & Finding the Data

Within the run's phase directory (e.g. `run-XYZ/plan-ingestion/`), Kaizen outputs are split into two levels:

1. **Aggregated Phase Telemetry**:
   - `kaizen-metrics.json`: High-level metrics for the phase run.
   - `kaizen-analysis.log`: Execution logs for the post-mortem analysis scripts.
   - `kaizen-suggestions.json`: Automatically generated prompt improvement suggestions.
   - `kaizen-correlation.md` / `kaizen-trends.md`: Markdown reports on prompt characteristics vs success metrics.

2. **Per-Feature Prompt Data** (`kaizen-prompts/`):
   Contains subdirectories for each feature generated (e.g. `standalone/PI-005/`), governed by the `KAIZEN_RUN_ID` and feature ID.

## 3. Reading the Per-Feature Prompts

Inside `kaizen-prompts/standalone/<FEATURE_ID>/`, you will find the exact artifacts sent to and received from the LLM.

### Metadata

- **`metadata.json`**: Documents the exact agent models used, the target files, boolean flags (like `has_existing_files`), and the full list of context keys that were active during prompt generation.

### The Prompt & Response Pairs

The Lead Contractor workflow runs three main phases per feature. You'll see corresponding files for each:

1. **Spec Phase** (`spec_user_prompt.md`, `spec_system_prompt.md`)
2. **Draft Phase** (`draft_user_prompt.md`, `draft_system_prompt.md`)
3. **Review Phase** (`review_user_prompt.md`, `review_system_prompt.md`)

Each phase also generates a captured LLM output:

- **`{phase}_response.md`**: The raw text output from the LLM. *Any configured PII redactions are applied before this file is written.*
- **`{phase}_response.meta.json`**: Describes if the LLM output exceeded the 2 MiB capture threshold and got truncated.

> [!WARNING]
> **Missing Response Files**: If you see user/system prompts but **no** response file (e.g., `draft_response.md` is missing), this may indicate either: (a) the LLM generation process suffered a hard fault (network timeout, empty response, syntax crash), or (b) the code generator did not forward raw responses via `GenerationResult.metadata` keys (`spec_raw_response`, `draft_raw_response`, `review_raw_response`). As of v0.5.0, the `LeadContractorGenerator` populates these keys from the workflow summary.

## 4. Analyzing Aggregated Metrics

### `kaizen-metrics.json`

This is the core health check of the run. Look for:

- `success_rate`: The ratio of PASS features to total features.
- `top_root_causes`: If failures occurred, what layer threw the block? (e.g., `generation_error`, `repair_exhausted`, `validation_failure`).
- `pipeline_attribution`: Locates exactly where in the pipeline the fault happened (e.g. `ollama_generation`).

### `kaizen-correlation.md`

This markdown file maps scalar characteristics (like feature sizes, prompt word counts, number of dependencies) to the PASS/FAIL groups.

- Look at the `PASS mean` vs `FAIL mean` columns.
- If the `FAIL mean` for `target_file_count` is consistently much higher than the `PASS mean`, it indicates your Micro Prime decomposer is failing to split large chunks of work effectively.

### `kaizen-suggestions.json`

Consolidated lessons-learned generated by the analysis engine. If Kaizen notices repeated failures heavily correlated with certain prompt structures (e.g., missing imports or missing return types), it will template a suggestion here to be injected into the next run via `--kaizen-config`.

## 5. Draft vs Disk: The Assembly Gap

> [!IMPORTANT]
> **The LLM draft and the file on disk can diverge dramatically.** Always compare both when investigating failures.

The Kaizen prompt data captures the LLM’s raw output (`draft_response.md`), but the final file on disk passes through several transformation steps that can corrupt or lose content:

1. **Skeleton splicing** (micro-prime): Element-level generated code is spliced into a skeleton. If the splice fails or the cache hit has no code, `raise NotImplementedError` stubs remain.
2. **AST merge** (integration): When `has_existing_files: true`, Python files are AST-merged with the existing target. If the source is a complete standalone file, additive merge keeps stale target definitions and duplicates "other" nodes (`__main__` guards, constants).
3. **File path resolution** (integration): Fallback path derivation can write files to the wrong location (e.g., project root instead of `src/`).

### How to check

```bash
# Compare draft artifact to file on disk
diff <(cat kaizen-prompts/standalone/PI-XXX/draft_response.md) <(cat src/path/to/file.py)

# Verify final file parses
python3 -c "import ast; ast.parse(open(‘src/path/to/file.py’).read())"

# Check for remaining stubs
grep -n "raise NotImplementedError" src/path/to/file.py
```

### Merge behavior by file type

| File type | `has_existing_files: true` | `has_existing_files: false` |
|-----------|---------------------------|----------------------------|
| Python (`.py`) | AST merge (additive → auto-replace if >50% overlap) | Direct copy |
| HTML, other | Direct copy (`can_merge` returns `False`) | Direct copy |

## 6. Micro-Prime Artifacts

When a feature routes through micro-prime (local Ollama), the telemetry structure differs from the Lead Contractor pattern. Micro-prime operates at the **element level** (individual functions/methods), not the file level.

### What to look for

- **Element results**: Each element has `success`, `tier`, `code`, `verification_verdict`, and optionally `escalation` with an `EscalationReason`.
- **Cache hits**: Elements with `verification_verdict: "skipped"` were served from `_success_cache`. If `code` is `null`, the cache hit carried no generated code — the skeleton stub remains.
- **Filled skeleton**: The `filled_skeleton` field on the file result shows the post-splice state. If it still contains `raise NotImplementedError`, splicing failed for one or more elements.
- **Escalation chain**: Elements can escalate from local (Ollama) → cloud fallback. Check the `escalation.reason` field for `STRUCTURAL_MISMATCH`, `REPAIR_EXHAUSTED`, etc.

### Common micro-prime failure patterns

| Pattern | Symptom | Where to look |
|---------|---------|---------------|
| Broken skeleton | File has `raise NotImplementedError` stubs | `filled_skeleton` field; element `code` values |
| Nested duplicate | Function defined inside itself | `draft_response.md`; repair step `over_generation_trim` |
| Wrong function | Generated code implements a different API | `draft_user_prompt.md` — was the element prompt specific enough? |
| Cache without code | `verification_verdict: "skipped"`, `code: null` | `_success_cache` was a `set` (fixed: now `dict` carrying code) |

## 7. Success Reporting Caveats

> [!WARNING]
> **`success: true` does not guarantee the file on disk is correct.** Always validate independently.

Success is determined by:
1. **Element-level verdict** — checks each element in isolation (can pass even when the assembled file is broken)
2. **Review score** — evaluates the LLM draft, not the final file on disk (a 97/100 PASS review can correspond to a garbled file)
3. **Post-assembly gate** — checks for remaining `raise NotImplementedError` stubs and escalates or marks incomplete

Known gaps in success reporting:
- Review scores reflect draft quality, not assembly quality
- A file can be "partially filled" (some stubs replaced, others remaining) and still report element-level successes
- AST merge conflicts are logged as warnings but don’t flip `success` to `false`

### Validation checklist for "successful" features

1. Does the file on disk parse? (`ast.parse()` for Python)
2. Are there remaining `raise NotImplementedError` stubs?
3. Does `diff` between draft and disk show unexpected additions/deletions?
4. For `has_existing_files` features: are there duplicate `__main__` guards, duplicate constants, or duplicate class definitions?

## 8. Cross-Feature Comparison

When multiple features target similar code (e.g., PI-001 and PI-002 both implement "Shared JSON Logger" for different services), compare outcomes to identify systemic vs feature-specific issues.

### What to compare

| Dimension | What it reveals |
|-----------|-----------------|
| Route (micro-prime vs cloud fallback) | Cloud fallback consistently produces better output; micro-prime failures are often structural |
| Cost ($0.00 vs $0.08+) | $0.00 means micro-prime only (no cloud call); non-zero means cloud fallback was triggered |
| Review score | Identical specs with divergent scores point to assembly issues, not LLM quality |
| File on disk | Side-by-side diff reveals whether the LLM produced equivalent code and assembly diverged |

### Example: Run 004 PI-001 vs PI-002

Both features had identical specs ("Shared JSON Logger") for different services. PI-001 routed through cloud fallback → correct file, 97/100. PI-002 stayed micro-prime only → broken skeleton, 97/100. The review scores were identical because both evaluated the *draft*, not the disk. The divergence was entirely in assembly.

## 9. Typical Debugging Workflow

1. You notice a feature failed during a run.
2. Open `kaizen-metrics.json` to identify the `top_root_causes`.
3. If it’s a generation fault, navigate to `kaizen-prompts/standalone/<FEATURE_ID>/`.
4. Inspect `draft_response.md` to see exactly what the LLM produced.
5. **Compare draft to file on disk** — `diff` the draft against the actual generated file to catch assembly/merge corruption.
6. If the file has stubs or garbled content, check the element results and `filled_skeleton` for micro-prime features, or the merge log for `has_existing_files` features.
7. If the LLM hallucinated, open `draft_user_prompt.md` to cross-reference what instructions it was given. Was the necessary context missing?
8. For cross-feature issues, compare sibling features that share specs to isolate route-specific vs prompt-specific failures.
9. Create an experimental hint in your `kaizen-config.json` targeting that feature, rerun with `--kaizen-config`, and observe if the new hint corrects the LLM behavior.
