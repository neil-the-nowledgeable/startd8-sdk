**Resolved open questions:**
- **OQ-1 ‚Üí Self-contained HTML file** with embedded JSON; generated by a `startd8 consult web <id>` command. No server.
- **OQ-2 ‚Üí Image indicators only in v1** (filename + short hash). Optional `--embed-images` (base64 from `source_path` at render time) is deferred.

### 0.1 Lessons-Learned Hardening (v0.3)

> Consulted the SDK lessons base, `CLAUDE.md`, and the consultation feature's own posture.

- **[Phantom-reference audit]** ‚Äî verified every symbol named here exists: `ConsultationStore.load`,
  `ConsultationSession` (`prompt`/`roster`/`turns_by_model`/`images`), `Turn`
  (`role`/`text`/`status`/`error`/tokens), `comparison_text`/`comparison_table` in
  `consultation/view.py`. See ¬ß6 Reference Audit.
- **[Overloaded-term co-location]** ‚Äî the web renderer is a sibling of the existing renderers:
  `render_html()` lives in `consultation/view.py` next to `comparison_text`/`comparison_table`, not
  a competing module (FR-WUI-7).
- **[Single-source vocabulary ownership]** ‚Äî the HTML reads the `ConsultationSession` schema owned by
  `consultation/models.py`; this doc does not restate field types (cites ¬ß6).
- **[Security posture inheritance]** ‚Äî the consultation feature deliberately keeps image **bytes and
  absolute paths out of persisted/exposed artifacts** (FR-MMC-5/6a). The HTML artifact inherits that:
  escape untrusted model text (XSS), show image **filenames** not absolute paths (FR-WUI-9).
- **[Prune phantom scope]** ‚Äî "send a follow-up from the web page" moved to a Non-Requirement (NR-2):
  it would make the artifact a mutating client, which contradicts "simple read snapshot" and the
  CLI-as-sole-writer posture.

## 1. Problem Statement

A consultation produces N models' answers (and follow-up threads), persisted to
`.startd8/consultations/<id>/session.json`. Today they're readable only as a TUI table
(`comparison_table`) or CLI text (`comparison_text`) ‚Äî a **flat, latest-answer-per-model** view.
There is no way to see each model's **full conversation** side-by-side, or to **collapse** the
models you're not focused on so you can compare two threads without the others' noise.

| Component | Current State | Gap |
|-----------|---------------|-----|
| Session data | Persisted `ConsultationSession` (full per-model threads) | Rich; underused by the flat views |
| TUI/CLI view | `comparison_table`/`comparison_text` ‚Äî latest answer per model | No full threads, no collapse, terminal-bound |
| Web view | none | **This doc** ‚Äî a simple, offline HTML snapshot |

---

## 2. Requirements

- **FR-WUI-1 ‚Äî Self-contained HTML snapshot.** `startd8 consult web <session-id>` generates a
  single `.html` file (into the session dir; `--open` to launch the browser) with the session data
  **embedded inline** ‚Äî it opens offline, no server and no network.
- **FR-WUI-2 ‚Äî All models at once.** The page shows **every** model's conversation thread
  simultaneously, side-by-side (responsive columns/panels).
- **FR-WUI-3 ‚Äî Collapse/expand per model.** Each model panel is **independently collapsible** ‚Äî the
  user can collapse one or more models' conversations (and expand them again). Include
  collapse-all / expand-all controls. Collapsed state is a pure client-side toggle.
- **FR-WUI-4 ‚Äî Full conversation, not just the latest.** Each panel renders the model's **ordered
  thread** (user prompt turns + assistant answers across follow-ups), role-styled, with the answer
  text rendered as markdown.
- **FR-WUI-5 ‚Äî Status + usage.** Per-model status (`ok` / `failed` / `skipped-non-vision`) and
  per-turn usage (input/output tokens, latency) are shown as badges; failed/skipped answers are
  visually distinct (and show the structured error for failures).
- **FR-WUI-6 ‚Äî Session header.** A header shows the session id, the original prompt, the roster, the
  image **indicators** (filename + short hash), and created/updated timestamps.
- **FR-WUI-7 ‚Äî Shared core, no logic fork.** The HTML is produced by a pure `render_html(session)`
  in `consultation/view.py` (beside `comparison_text`/`comparison_table`); the `web` command loads
  the session via the existing `ConsultationStore`/`ConsultationService`. No new data path.
- **FR-WUI-8 ‚Äî Responsive + accessible.** Panels stack on narrow screens; collapse toggles are
  keyboard-operable and screen-reader-labeled (built with `/frontend-design`; apply
  `/universal-design` for WCAG basics ‚Äî semantic `<details>`/buttons, focus, contrast).
- **FR-WUI-9 ‚Äî Safe embedding (security).** All model-generated text is **escaped/sanitized** before
  embedding (untrusted output ‚Üí no XSS); image references show **filename + short hash only**, never
  the absolute `source_path` (inherits FR-MMC-5/6a redaction posture).
- **FR-WUI-10 ‚Äî Follow-up command composer (leverage the CLI).** The page includes a composer ‚Äî a
  follow-up textarea + a target selector (**all models** or one model) ‚Äî that generates the exact
  `startd8 consult reply <id> --to <target> --prompt <‚Ä¶>` command for the user to copy and run. The
  page **never executes** it (a `file://` page is sandboxed and cannot; NR-2 holds): it only *builds*
  the command via `textContent`. The free-text prompt is **POSIX shell-quoted** (single-quote wrap;
  `'` ‚Üí `'\''`) so any characters (`$( )`, backticks, `;`, quotes, newlines) round-trip byte-identical
  with no command substitution/injection. After running it, the user re-renders via `consult web`.

## 3. Non-Requirements

- **NR-1 ‚Äî Not a live/refreshing server.** A static point-in-time snapshot; re-run `web` after new
  turns. No websocket, no polling, no `startd8 serve` dependency.
- **NR-2 ‚Äî Read-only (does not execute or mutate).** The page cannot itself send prompts, run
  follow-ups, retry, or mutate the session ‚Äî a `file://` page is browser-sandboxed. It **may compose**
  a `consult reply` command for the user to run in their terminal (FR-WUI-10), but execution and all
  writes stay in the CLI/TUI (the single-writer posture holds). No server, no eval, no shell-out.
- **NR-3 ‚Äî No hosting / auth / sharing.** A local file the user opens; not served or access-controlled.
- **NR-4 ‚Äî No image pixels in v1.** Bytes aren't persisted (FR-MMC-6a); indicators only. Optional
  `--embed-images` is deferred.
- **NR-5 ‚Äî No new dependency / framework.** Plain HTML/CSS/JS artifact (no React build, no CDN
  fetch); nothing added to the workflow server.

## 4. First Acceptance Scenario

Using the live-smoke session (`startd8 consult web <that-session-id> --open`):
1. The browser opens a page showing **all three** models (claude-opus-4-8, gpt-5.5, gemini-2.5-pro)
   side-by-side, each with its full door-repair thread (initial answer + the Gemini follow-up).
2. The user **collapses** claude and gpt-5.5; only the Gemini thread remains expanded, readable
   without the others' noise. Expanding them restores the side-by-side view.
3. Each panel shows status = ok and per-turn token/latency badges; the header shows the prompt and
   the two image indicators (`first-picture.jpeg`, `second-picture.jpeg`).

## 5. Open Questions

- **OQ-4 ‚Äî Output location.** Write the `.html` into `.startd8/consultations/<id>/view.html`
  (alongside `session.json`/`summary.md`) ‚Äî confirm vs a `--out` override.
- **OQ-5 ‚Äî Markdown renderer.** Vendored-inline JS (e.g. a tiny `marked`-like) vs server-side
  pre-render to sanitized HTML at generate time. Both must be dependency-free/offline.
- **OQ-6 ‚Üí RESOLVED: all expanded by default.** The page opens with every model's conversation
  visible side-by-side ("see all at once"); the user collapses what they don't want. Collapse-all /
  expand-all controls remain (FR-WUI-3).

## 7. Alternatives Considered (do not re-litigate)

**Native deterministic ($0) generation ‚Äî REJECTED (wrong shape, verified 2026-07-03).** The SDK's
deterministic generators (`backend_codegen`, `view_codegen`, `frontend_codegen`, `presentation_polish`)
all take a **`.prisma` relational-entity contract** and emit CRUD apps / entity composite-views /
Zod schema / CSS restyle. The consultation view is a **read-only visualization of a nested JSON
document** (`turns_by_model: dict[model_id ‚Üí list[Turn]]`), not relational entities:
- `view_codegen` board/workspace bucket *one entity's rows* into columns ‚Üí flat `<ul>` of record
  ids (`renderers.py:1300-1314`); `rendered-content` is single-entity single-field pseudo-markdown
  (`renderers.py:420-474`). No per-model conversation columns, no collapse, no real markdown.
- `backend_codegen` = list/detail/create/edit CRUD per entity; its only non-entity page renders an
  authored `.md` **file**, not a data document.
- `presentation_polish` = CSS restyle of an already-generated app; cannot emit a page.

Forcing it through would require modeling `ConsultationSession` as prisma entities (a second
persistence model diverging from the JSON store) and would still not yield the columnar-collapsible-
chat UX. A ~one-function bespoke `render_html` over the existing Pydantic model is both simpler and
the correct fit. This is consistent with `CLAUDE.md`'s bucket separation: the deterministic path is
for **applicational completion of CRUD apps from a data contract**, not one-off visualizations.
