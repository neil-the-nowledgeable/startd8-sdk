# CRP Focus — Synthetic-Probe SLI P0 (#308 P0, v0.3.1)

**Least-reviewed artifact:** this entire requirements doc — brand new, zero prior external review.

## Weight the review on these concerns

1. **FR-5 `pending_probes` vs dead-SLI distinction (highest priority).** P0 emits a freshness SLO whose
   metric doesn't exist until a runner runs (P1/P2). It records a `pending_probes` class so `compare`
   doesn't false-flag it as a #274 dead SLI. Is that distinction actually enforced end-to-end? If P0 also
   writes an SLO YAML with a `target` (FR-4 graded case), does `compare-live` (even though NR-3 defers the
   Tier-B verdict) still replay it and red-flag it? Is there a leak where a pending SLI reaches the dead-SLI
   classifier? Should the SLO file even be written in the graded case, or only the pending_probes record?

2. **FR-3 published-metric contract single-sourcing.** The P0 SLI query and the future P1 runner must emit
   the SAME metric string or the SLI never binds. Is deriving `probe_<name>_seconds` as a default robust
   (name collisions across probes/services, label dimensions, unit)? Should the contract live on a
   descriptor profile (like #307 FR-3) rather than a bare default string? How does `metric_kind`
   (gauge/histogram) map to the query shape, and what if it's wrong?

3. **FR-1 `DeclaredProbe` model + the `assert_` field naming.** `assert` is a Python keyword — the field is
   `assert_`. Is that the right carry shape? Are `action`/`poll` opaque strings enough for P0, or do they
   need structure now to avoid a breaking change at P1? Is `signal_kind` defaulting to `freshness` right
   (OQ-1)?

4. **FR-6 compare.py + the fourth lane.** This adds a FOURTH positive-finding surface (base/functional/span
   bindings + now pending_probes) to `ComparisonReport`/`build`/`render`. Is the compare surface scaling, or
   does the accumulation of `bound_*`/`pending_*` fields argue for a unified positive-findings shape (echoing
   the #307 R1-F10 single-authority concern)?

5. **P0/P1 boundary integrity.** Does P0 genuinely stand alone and deliver value with NO runtime, or is
   there a hidden dependency that makes the emitted SLO meaningless until P1? Is the "positive finding"
   claim honest for a static artifact whose metric is unpopulated?
