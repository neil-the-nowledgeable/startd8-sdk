# CRP Focus — Declared-Series Functional SLI Binding (v0.3.1)

**Least-reviewed artifact:** this entire requirements doc — brand new, zero prior external review.

## Weight the review on these four concerns

1. **OQ-5 — threshold-deferred artifact shape (highest priority).** When a declared series covers a
   functional kind but carries no `target`, should the SDK emit (a) a target-less/annotated SLO YAML on
   disk, or (b) keep it in the `deferred_declared_kinds` gap record *enriched with the grounded query*?
   The spec leans (b). Pressure-test both: on-disk consumers (OpenSLO validators, `compare-live` replay,
   dashboards) vs. discoverability of a gap-only record. Recommend a decision.

2. **New optional `target` on the shared `DeclaredEmittedSeries` model.** Schema/contract impact: does
   adding a per-series target create ambiguity with base-RED sourcing (manifest/importance default)?
   Parsing/validation (`_parse_declared_series`)? Back-compat (absent ⇒ byte-identical)? Naming.

3. **New on-disk artifact lane `{svc}-declared-functional-slo.yaml` + `bound_declared_functional`
   fr_coverage key.** Downstream contract impact: `compare-live` baselines (new replayable SLIs → new
   verdict ids), the `compare.py` gap-class table, dashboards keyed on `fr_coverage`. Is a separate lane
   right, or should it fold into the existing functional-slo doc?

4. **FR-7 FR-precedence de-dup correctness.** If a `functional_requirements` FR and a declared series
   both cover the same `(service, signal_kind)`, the FR wins. Is the de-dup key correct? Race/ordering
   across the two generators? Is the skip observably recorded?

## Settled — do NOT relitigate

- The #300 A–D fixes are shipped and out of scope for this review.
- **NR-1 (no fabricated/default thresholds).** Firm design-principle decision (Genchi Genbutsu). Do not
  propose a default-threshold table or importance-scaled functional thresholds.
- The reflective-loop discovery that the *query* is determinable but the *target* is not — accepted premise.
