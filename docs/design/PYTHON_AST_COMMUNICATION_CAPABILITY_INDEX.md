## 1. Purpose

This index is a **numbered catalog** of Python language structure (via the stdlib `ast` module) cross-walked to the **application communication patterns** OpenTelemetry standardizes in landscape §5.

It answers two questions for code observability and instrumentation-contract derivation:

1. **What can Python express structurally?** — the complete `ast` node taxonomy + manifest-extractable elements.
2. **Which OTel communication domains does that structure imply?** — HTTP, RPC, messaging, database, GraphQL, FaaS, feature flags, GenAI, CI/CD, CLI, DNS, object stores, cloud SDKs.

The index is the **static half** of the map; runtime OTel spans/metrics are the **dynamic half**. AST tells you *where* communication *could* occur; instrumentation tells you *what actually happened*.

---

## 2. Is AST the right mechanism?

**For Python: yes — as the primary static substrate.**

| Criterion | Python `ast` | Alternatives |
| --- | --- | --- |
| **Completeness** | Full syntax tree for parsed source; stdlib, no build step | Regex parsers (~80–90% recall in other StartD8 langs) |
| **Determinism** | Same source → same tree (modulo `python_version`) | tree-sitter (multi-lang, not Python-semantics-native) |
| **Communication detection** | Imports + call sites + decorators are AST nodes | Type checkers (mypy/pyright) — heavier, optional Phase 2 |
| **Inner-loop cost** | Milliseconds per file | CodeQL — minutes, needs buildable whole program |
| **Cross-file limits** | Single-file only; import strings are hints not resolved targets | SCIP/Jedi — cross-file resolution (Phase 2) |

**Caveats (explicit):**

- AST indexes **syntax**, not **runtime behavior**. A `Call` node does not prove a network request occurred.
- Import strings are **unresolved** until a resolver runs (`importlib`, Jedi, SCIP).
- Generated / dynamic code (`exec`, metaclass magic) may lack static nodes.
- OTel **metrics** and **span attributes** are not AST nodes — they appear in the crosswalk as *downstream signals* linked by instrumentation libraries.

StartD8 already treats Python AST as **authoritative** for manifest generation (`TIER_AUTHORITATIVE` in `manifest_adapter.py`). This catalog formalizes that into a numbered, OTel-aligned index.

## 3. Mathematical map (formal sketch)

Let:

- **N** = `{ PY-AST-001 … PY-AST-132 }` — stdlib `ast` node types (see `ast-nodes.json`)
- **M** = `{ PY-MAN-001 … PY-MAN-009 }` — manifest `ElementKind` extractors
- **L** = `{ PY-LC-001 … PY-LC-010 }` — composite language capabilities (subgraphs)
- **C** = `{ PY-OTEL-5.* }` — OTel landscape §5 communication patterns (15 entries)
- **Σ** = finite alphabet of import/call/decorator signature strings

Define detectors:

```
detect_import : Source → P(Σ)     # Import/ImportFrom module names
detect_call   : Source → P(Σ)     # Call attribute/name suffixes
detect_decor  : Source → P(Σ)     # decorator_list names
```

Crosswalk function (partial, multi-valued):

```
φ : C → P(N) × P(Σ)
φ(pattern) = (relevant_ast_nodes, import ∪ call ∪ decorator signatures)
```

Static communication **hypothesis** for file `f`:

```
hyp(f) = { c ∈ C | detect_import(f) ∩ φ(c).Σ ≠ ∅
              ∨ detect_call(f) ∩ φ(c).Σ ≠ ∅
              ∨ detect_decor(f) ∩ φ(c).Σ ≠ ∅ }
```

Runtime OTel evidence refines `hyp(f)` → confirmed spans. Tier 0 `coverage-attestation.json` validates the backend side; this index validates the **static detectability** side for Python.

## 9. Using the index

### 9.1 Code observability pipeline

```
Python source
    → ast.parse → PY-AST-* nodes
    → generate_file_manifest → PY-MAN-* elements
    → detect_import/call/decor → φ crosswalk → PY-OTEL-5.* hypotheses
    → (optional) runtime OTel spans → confirm/refute
```

### 9.2 Instrumentation contract derivation

Use `communication-crosswalk.json` `attribute_hints` and `metric_hints` as the **static allowlist** of semconv keys a Python file may participate in — feeds ContextCore `_PROTOCOL_METRICS` extension (see [derivation-handoff.md](./otel-demo-corpus/derivation-handoff.md)).

### 9.3 Benchmark / corpus analysis

When scoring generated Python (e.g. OTel Demo `recommendation`, `product-reviews`), check:

1. Required PY-MAN elements present (RPC handler)
2. φ patterns match service role
3. No forbidden cross-corpus imports (contamination firewall)

#### 9.3.1 Resolver pass (`hyp(f)`)

The static resolver implements §3 formally:

```
hyp(f) = { c ∈ C | detect_import/call/decor intersect φ(c) signatures }
```

| Script | Role |
| --- | --- |
| `scripts/python_capability_resolver.py` | Core: `extract_signals`, `match_patterns`, `analyze_corpus` |
| `scripts/analyze_otel_demo_python_coverage.py` | CLI: walk OTel Demo `src/**/*.py`, emit JSON + markdown |

```bash
# Ensure clone (tag 2.2.0)
scripts/otel_demo/bring_up.sh --no-clone

python3 scripts/analyze_otel_demo_python_coverage.py
# → docs/design/python-capability-index/otel-demo-python-coverage.{json,md}
```

**Coverage model** — four dimensions, each `detected / catalog × 100`:

| Dimension | Catalog | Detected when |
| --- | --- | --- |
| `communication_patterns` | 15 `PY-OTEL-5.*` | import/call/decorator signature match |
| `ast_nodes` | 132 `PY-AST-*` | node type appears in corpus AST walk |
| `language_composites` | 10 `PY-LC-*` | composite ast_nodes (+ decorators for LC-005) |
| `manifest_kinds` | 9 `PY-MAN-*` | ElementKind extractable from AST |

**Overall index %** = arithmetic mean of the four dimension percentages.

Generated `*_pb2.py` / `*_pb2_grpc.py` are skipped by default (`--include-generated` to retain).
