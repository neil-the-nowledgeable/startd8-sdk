## 4. Remaining Issues

### OPEN: Micro-prime-only features with all elements "passing" still produce broken files

**Affected:** PI-004, PI-006 (and likely any future `micro_prime_only: true` feature)

**Mechanism:** The INV-6 stub detection gate checks whether stubs remain *after* element-level verification. But if `verification_verdict: "pass"` for all elements, the gate never fires. The problem is:
1. `over_generation_trim` removes excess AST nodes (imports, nested def) but the trimmed result is re-wrapped by `bare_statement_wrap`
2. The element-level AST check passes (the code is syntactically valid in isolation)
3. But the **assembled file** still contains the skeleton structure with `[STARTD8-SKELETON]` markers

**Fix needed:** Post-assembly file-level validation that checks:
- No `[STARTD8-SKELETON]` markers remain
- No nested duplicate function definitions (function defined inside a function with the same name)
- The file implements the spec's required functions/classes (not just any AST-valid code)

### OPEN: Ollama `startd8-coder` over-generation pattern (INV-7)

Every Ollama generation still produces nested duplicates:

```
def foo() -> None:
    import bar
    def foo() -> None:     # <-- re-emitted signature
        actual_body()
```

The `over_generation_trim` repair step trims the inner imports but the nested `def` persists. This is a model behavior issue — the prompt should be adjusted to make clear the model should output **only the function body**, not re-emit the signature.

### OPEN: Cross-feature divergence (INV-9)

PI-001 and PI-002 are still functionally different despite "identical copy" spec. The review doesn't have cross-feature context.
