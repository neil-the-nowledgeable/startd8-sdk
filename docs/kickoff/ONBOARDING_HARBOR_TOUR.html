⚓ Onboarding & Kickoff
A harbor tour of how a directory becomes a StartD8 project
StartD8 SDK · main deterministic · $0 · no-LLM core generated 2026-07-03

Where a project comes ashore. This tour walks the onboarding harbor: the deterministic $0 path that turns any directory into a set-up StartD8 project, opts it into the VIPP proposal loop, and hands it off to the interactive kickoff surfaces. Every berth below is a real capability on main; the arrows between them are the seams that actually connect.

The organizing idea (requirements §0): project ground-truth adjudicates, it never originates. So onboarding never invents content — a healthy project is left inbox-ready, and production is always gated on a real, declared gap.

The voyage — one directory, end to end

1

Make it a StartD8 project

project init detects the shape (greenfield / brownfield_ready / brownfield_partial) from cheap on-disk presence checks, establishes the .startd8/vipp posting, and stands up the inbox mechanism — leaving the project inbox-ready. Re-runnable as a clean no-op.

# deterministic, $0 startd8 project init .
2

Produce an inbox — only on a real gap

The $0 non-interactive producer seam (closes #76): serialize an authored proposal set, or, on a greenfield dir, the one deterministic instantiate proposal. Every entry is validated per-kind before anything is written — a bad entry fails with exit 2 and no half-written inbox.

startd8 project init . --proposals proposals.yaml # or: --instantiate
3

Negotiate the inbox against ground truth

VIPP adjudicates each proposal against the project's Sapper ground truth ($0, deterministic) into source-labeled dispositions. On a healthy project with no pending inbox this is now a clean exit 0 — "inbox-ready; nothing to negotiate yet" (the OQ-8 fix).

startd8 vipp negotiate --project-root .
4

Apply the accepted proposals — at your privilege

Preview by default; --apply writes accepted proposals through the human-confirm floor, provenance-pinned to the trusted inbox, then shreds the inbox once consumed.

startd8 vipp apply --project-root . # --apply to write

⚓ The trust boundary is a file

Producers and the negotiator never share memory — the inbox .startd8/vipp/proposals-inbox.json is the boundary (out-of-process only). Its scaffold (.gitignore + monotonic inbox-seq) is stood up by the single shared vipp_seam.ensure_inbox_scaffold, so the producer path and the inbox-ready path can never drift. A symlinked project root is rejected at the source (vipp.context.ensure_posting confinement).

The berths — who's docked in the harbor

project init

Deterministic onboarding · $0 · CLI

A thin orchestrator composing already-confined seams — no new write primitive. The front door.

  • detect_shape — cheap on-disk triage (contract, app/, the shared kickoff domains)
  • establish_postings — VIPP always, FDE opt-in
  • ready_inbox / produce_inbox — mechanism, then gated production
  • --check drift audit (0=in-sync / 1=drift / 2=error)
📄 src/startd8/project/init.py

VIPP

Project-side negotiator / applier

The OBSERVED(project)-authority dual of the FDE. Receives the inbox, adjudicates against ground truth, applies accepted proposals at project privilege.

  • negotiate → source-labeled dispositions ($0)
  • apply → provenance-pinned, human-confirm floor, shred-on-done
  • mints only OBSERVED(project) claims — cannot forge SDK authority
📄 src/startd8/vipp/ · kickoff_experience/vipp_seam.py

Concierge

Onboarding host · assist, not operate

The human + agent front door to onboarding. Read-only survey / assess; CLI-gated writes; an agentic chat whose leftovers can serialize to the same VIPP inbox.

  • survey / assess — brownfield triage + $0-cascade readiness
  • instantiate-kickoff / log-friction
  • chat session-end → maybe_serialize_buffer (the one host inbox producer)
📄 src/startd8/concierge/ · cli_kickoff.py

Red Carpet

Prescriptive conductor · pure, read-only

Projects assessment + on-disk manifests into a staged-build playbook (advisories + ranked next-steps). Applies proposals directly at human privilege — never a gate.

  • stages: data_model → manifests → value_inputs → content → run
  • points at the Manifest Suggester at the "which screens?" moment
  • note: applies in-process; does not serialize to the VIPP inbox
📄 src/startd8/kickoff_experience/red_carpet*.py

Kickoff Experience

Interactive surfaces · web + TUI + MCP

A deterministic front-end over the manifest-extraction grammar, sharing one canonical view-model across surfaces. Writes the four kickoff input YAMLs the rest of the harbor reads.

  • canonical view-model (state.py) + readiness (readiness.py)
  • SDK-internal step/field config (manifest.py — WriteTargets)
  • web app, TUI Concierge, read-only kickoff-state MCP tool
📄 src/startd8/kickoff_experience/

Manifest Suggester

Screen drafting · $0 core + opt-in persona pass

Drafts candidate screens (composite views + non-entity pages) grounded in the schema + stakeholder roster — the "which screens?" gap. Human approves; the CLI is the sole writer.

  • startd8 screens suggest / review / approve / reject
  • $0 baseline dashboard + opt-in designer-persona drafts
  • writes views.yaml / pages.yaml via the manifest proposal kind
📄 src/startd8/manifest_suggester/ · cli_screens.py

The harbor master — one validation channel

⚓ build_proposal — every producer validates through one seam

Producing a VIPP inbox is irreducibly: validate a proposal per its kind → construct a typed action → serialize the confined, parity-guarded envelope. Serialization was already one function (serialize_buffer); validation is now one too — build_proposal(args) → ProposedAction, a pure primitive that raises a typed error and has no buffer / telemetry / message side effects. The agentic make_propose_handler and the deterministic project init producer both validate through it — so the three producers (agentic · deterministic · authored) stay distinct at the entry, but share exactly two primitives underneath.

The channel map — what actually connects

From the integration audit. The mechanical seams are clean; a few orchestration seams between capabilities that grew up in parallel are still open (by design, not bug).

SeamHow it connectsVerdict
project init → VIPP inbox-ready → negotiate; shared ensure_inbox_scaffold + serialize_buffer (envelope parity); OQ-8 closes the negotiate UX cliff; confinement lives in ensure_posting itself. CLEAN
project init ↔ Concierge shares the one canonical KICKOFF_INPUT_DOMAINS constant + the confined safe-writer. Detection mirrors Concierge; it does not call the heavier assess. CLEAN
Concierge chat → inbox a second inbox producer (session-end maybe_serialize_buffer). Same build_proposal + serialize_buffer; entry points not unified (coexist via no-clobber-of-undrained). OVERLAP
Red Carpet → inbox applies proposals directly at human privilege; no VIPP handoff at REPL exit (asymmetric with Concierge chat). A feature decision, not complexity to remove. GAP
Manifest Suggester → inbox shares only the manifest apply primitive; applies straight to disk, bypassing the inbox. Its outputs aren't in detect_shape's signal set. build_proposal makes the seam trivial when wanted. GAP
Kickoff inputs contract the kickoff experience writes docs/kickoff/inputs/*.yaml; project init only reads them for shape. No shared "populate the inputs" call — a partial project drops back into the interactive surface. GAP
Welcome Mat design-doc only — superseded by Red Carpet; not implemented in code. N/A

⚓ Verdict

Onboarding plugs into VIPP cleanly and composes with Concierge at the mechanism level. It sits alongside Red Carpet, the Manifest Suggester, and the interactive kickoff rather than being woven into them — none of the open seams are bugs; they're missing connections between capabilities that grew up in parallel, and build_proposal now makes closing them cheap.
