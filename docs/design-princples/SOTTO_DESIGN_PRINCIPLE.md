# Sotto Design Principle

Purpose: establish a cross-cutting design principle for the startd8-sdk deterministic codegen
cascade — authored content (the *words* layer) must be added to generated output so gently that, when
the content is absent, the output is byte-identical to a build that never knew the feature existed.

This document is intentionally living guidance. Update it as new authored-content surfaces are added.

---

## The Principle

**Sotto** (そっと) — "softly, gently; without disturbing." In Japanese, doing something *sotto* means
acting so quietly that you leave the existing arrangement undisturbed — setting a cup down without a
sound, closing a door so it doesn't wake anyone.

Applied to the cascade: **a capability that lets a human author content over deterministically
generated code must bind that content through a hash-exempt, presence-gated seam. When the authored
input is absent, the owned output must be byte-identical to the pre-feature build. The feature speaks
only when invoked; otherwise it leaves no trace.**

The mechanism that satisfies this is the **prose-gated additive-manifest** pattern:

1. **Authored content lives in its own standalone manifest** — never inlined into an owned, hashed
   file. (`pages.yaml` page bodies, `view_prose.yaml` view copy, `ai_passes.yaml` prompt text.)
2. **The content renders to an *untracked, header-less* fragment** — a file with no generator
   provenance header, excluded from the drift hash. Editing the words never trips `--check`.
3. **The owned template gains the `{% include %}` (or runtime read) *only when the content is
   present*.** No manifest → no include emitted → the template is byte-for-byte what it was before the
   words layer existed.

The result: editing copy is a $0, drift-free, downstream-safe act. The structural skeleton (bucket 1)
stays deterministic and hashed; the words (bucket 2/4 authored text) ride alongside it without ever
disturbing it.

---

## The Diagnostic Question

> **"When the authored content is absent, is the owned output byte-identical to a build that never
> knew this feature existed?"**

If the answer is no — if merely *adding the capability* changed the generated bytes of an app that
doesn't use it — there is a Sotto violation. The feature is not additive; it is intrusive.

A second, sharper form for the words layer specifically:

> **"Can a human edit this content without tripping `--check` drift, and without that edit propagating
> into a downstream regeneration?"**

If editing the words re-hashes an owned file, the content was inlined where it should have been gated.

## Relationship to Other Principles

Sotto joins the anti-waste / anti-disruption family alongside the existing design principles:

| Principle | Japanese | Focus | What it protects | Diagnostic Question |
|-----------|----------|-------|------------------|---------------------|
| **Mottainai** | もったいない | Don't discard artifacts | Invested computation within a run | "Has this already been computed?" |
| **Kaizen** | 改善 | Don't discard lessons | Learning across runs | "Have we seen this problem before?" |
| **Warm Up** | ウォームアップ | Don't discard context | Knowledge across toolchain transitions | "Does the next tool know what the last learned?" |
| **Hayai** | 早い | Don't defer enforcement | Artifacts from contamination | "Do we know enough to prevent this defect?" |
| **Sotto** | そっと | **Don't disturb what exists** | The deterministic skeleton + downstream builds | "When the content is absent, is the output byte-identical?" |

Where Hayai asks "are we binding quality knowledge early enough?", Sotto asks the inverse-facing
question: **"are we binding *authored* content gently enough that it never disturbs the deterministic
output it rides on?"** Hayai protects the artifact from bad input; Sotto protects the artifact from the
feature itself.

Sotto is also the operational guarantee behind the **bucket separation** (CLAUDE.md): bucket 1 (the
deterministic application) must stay deterministic and hashed even as bucket 2/4 (authored words) is
layered on. The prose-gated seam is *how* the words layer stays out of bucket 1's hash.

## Why This Matters

### The drift problem

The cascade's correctness story rests on **idempotent, hashed generation**: `generate backend`
followed by `generate backend --check` must report `in_sync`, and a downstream consumer that
regenerates must see no surprise diffs. Owned files carry a provenance header and participate in the
drift hash.

If authored content were inlined into an owned file, three things break at once:

1. **Editing copy trips `--check`.** The author changes a sentence; the file's hash changes; the gate
   reports drift on a file the author legitimately edited. The author is now fighting the generator.
2. **Regeneration clobbers the edit.** The next `generate` re-emits the owned file from the contract,
   silently discarding the authored words.
3. **Downstream consumers inherit churn.** Every consumer that regenerates sees the owned file change,
   even if they never touched the words — surprise drift that reads as "the generator is unstable."

The prose-gated additive-manifest pattern dissolves all three: the words live in their own manifest,
render to an unhashed fragment, and the owned template only references them when present.

### Byte-identical-when-absent is the testable contract

The load-bearing guarantee is **byte-identical-when-absent**, and it is cheaply testable: generate an
app with no authored content, generate the same app after the words capability ships, and diff the
owned output. Zero diff = Sotto satisfied. This is the regression lock every authored-content surface
must carry (FR-DRC-5 / the skip-hook recognition tests are this lock for the drift seam).

## The Pattern: Prose-Gated Additive Manifest

The Sotto-compliant recipe for any new authored-content surface:

1. **Give the content a standalone manifest.** A dedicated `*.yaml` (or `*.md`) keyed by the structural
   element it decorates (page slug, view name, AI-pass id). Never a new field on an owned, hashed file.
2. **Parse it loud-fail, strict.** Unknown keys / unknown targets are errors at compile time, not
   silent drops — the manifest is a contract (`parse_view_prose(known_views=…)` rejects copy for a view
   that doesn't exist).
3. **Render to an untracked, header-less fragment.** No provenance header → excluded from the drift
   hash by construction. The fragment is regenerated freely; the author's source of truth is the
   manifest, not the fragment.
4. **Gate the owned-template reference on presence.** Emit the `{% include %}` (generate-time) or the
   read (runtime) **only when the content exists**. Absent manifest ⇒ no reference ⇒ byte-identical
   owned output.
5. **Honor the archetype's escaping contract.** Authored text becomes served HTML. Each include site
   declares markdown-rendered vs auto-escaped vs raw; the producer must emit into the seam the consumer
   expects (R1-F9). Sotto is *gentle*, not *unsafe* — gating must not open an injection surface.
6. **Carry a byte-identical-when-absent regression test.** Prove that shipping the capability did not
   change the bytes of a no-content build.

## Instances in the SDK

| Surface | Manifest | Rendered to | Binding time | Escaping |
|---------|----------|-------------|--------------|----------|
| **Page bodies** | `pages.yaml` → `app/pages/*.md` | Untracked markdown fragment, `{% include %}`-d by the owned page template | **Generate-time** | Markdown-rendered |
| **View copy** (the words layer) | `view_prose.yaml` | Untracked header-less Jinja fragment, conditionally `{% include %}`-d by the owned view template | **Generate-time** | Per-archetype (escaped title/intro; outcome lines) |
| **AI-pass prompts** (*related variant*) | `ai_passes.yaml` prompt text | Read directly at request time (`ai_layer.py`) — **not a generate-time fragment** | **Runtime** | N/A (prompt text, not page HTML) |

The first two are the canonical **generate-time fragment** instances. The third — the AI-pass prompt —
is a **related runtime-binding variant**: it shares the "authored content lives in its own manifest,
absent ⇒ default behavior" spirit, but the content is *read at request time*, never materialized as an
untracked fragment. Its time-of-binding, format, and discovery all differ.

**Why these are documented, not refactored into one primitive (OQ-4 / dropped FR-DP-2):** the three
mechanisms are *intentionally divergent* — generate-time vs runtime binding, markdown vs escaped vs
raw, fragment-on-disk vs read-in-memory. A shared primitive would either leak all three axes as
parameters (no simplification) or force one mechanism onto a binding model that doesn't fit (partly
impossible). The **principle** is the reusable asset here; the implementations rightly differ.

## Checklist: Adding a New Authored-Content Surface

When you give humans a way to author content over generated code:

- [ ] **The content lives in its own manifest** — not a new field on an owned, hashed file.
- [ ] **The manifest parses strict / loud-fail** — unknown keys and dangling targets are compile-time
      errors, keyed to a known structural element.
- [ ] **The rendered output is untracked / header-less** — excluded from the drift hash by
      construction, so editing the words never trips `--check`.
- [ ] **The owned-template reference is presence-gated** — emitted only when the content exists.
- [ ] **A byte-identical-when-absent test exists** — proves shipping the capability changed nothing for
      a no-content build.
- [ ] **The escaping contract is honored at each include site** — authored text reaches served HTML
      through the seam the archetype declares (markdown / escaped / raw), never raw into a raw include.
- [ ] **The wireframe / pre-gen readout reports coverage** — which elements carry authored content vs
      render raw machine names (so absence is *visible*, not silent).

## Anti-Patterns (Sotto Violations)

| Violation | Signal | Why it's a violation |
|-----------|--------|----------------------|
| **Inlined copy** | Authored text added as a field on an owned, hashed file | Editing copy trips `--check`; regeneration clobbers the edit; downstream sees churn. |
| **Header on the fragment** | The rendered content fragment carries a provenance header | It re-enters the drift hash; editing the words now reports drift. |
| **Unconditional include** | The owned template always `{% include %}`s the fragment, empty or not | A no-content build is no longer byte-identical to the pre-feature build — the include itself is the diff. |
| **Silent drop** | A copy block for a non-existent target is ignored instead of erroring | The author's intent is lost without feedback; the manifest stops being a contract. |
| **Raw into raw** | Authored text emitted unescaped into a raw-include archetype | Gentle on drift, intrusive on security — an injection surface (R1-F9). |
